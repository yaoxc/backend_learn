# 用户下单：资金中心修改用户资金过程

本文档说明**资金服务**的实现：资金指令的输入与输出、处理流程、落库与执行的关系，以及如何保证**资金结算结果不丢、可恢复**；并记录改造过程，并给出从一条结算资金指令到资金处理结果的数据示例。

---

## 一、资金服务在整体链路中的位置

```
订单 → 撮合 → exchange-match-result
  → Market: 订单/钱包/流水落库 + 清算(clearing_result + exchange-clearing-result)
  → 结算: 消费 exchange-clearing-result → settlement_result + exchange-fund-instruction
  → 资金服务: FundInstructionConsumer 消费 exchange-fund-instruction
  → 落库 processed_fund_instruction → 按条执行钱包操作（member_wallet 加减可用/冻结）
```

资金服务消费结算产出的**资金指令**，不再发 Kafka，仅**落库执行记录 + 直接改 member_wallet**；幂等与重试依赖 **processed_fund_instruction** 表。

---

## 二、资金指令输入

- **来源**：Kafka topic **exchange-fund-instruction**（结算服务先落 settlement_result 再发该 topic）。
- **内容**：单条消息 key=messageId（与清算/结算一致），value=FundInstructionDTO JSON（messageId, symbol, ts, instructions）。每条 instruction 含 memberId, orderId, symbol, amount, type(INCOME/OUTCOME/FEE/REFUND/FEE_REVENUE)。
- **触发**：**FundInstructionConsumer** 订阅 exchange-fund-instruction，收到后调用 **FundInstructionExecuteService.processAndExecute(messageId, payload)**。
- **幂等**：若表 **processed_fund_instruction** 已存在该 messageId 且 status=PROCESSED，直接 return，不重复执行。

---

## 三、资金指令输出与执行

### 3.1 只落库 + 改钱包，不发 Kafka

| 输出 | 说明 |
|------|------|
| **落库** | 表 **processed_fund_instruction**（exchange-core 实体），字段：id, message_id(唯一), status(PENDING/PROCESSED/FAILED), payload(JSON 文本), created_at, processed_at。先插入 PENDING 并保存 payload，再按条执行；成功则更新 PROCESSED，失败则更新 FAILED。 |
| **钱包** | 对 **member_wallet** 的更新：INCOME → increaseBalance；OUTCOME → decreaseFrozen；FEE → decreaseBalance(deductBalance)；FEE_REVENUE → increaseBalance(平台钱包)；REFUND → thawBalance。 |
| **Kafka** | 资金服务**不再**向下游发 Kafka；下游若需感知资金变动，可查 member_wallet / member_transaction 或由其他链路推送。 |

### 3.2 指令类型与钱包操作对应

| 指令 type | amount 符号 | 钱包操作 |
|-----------|-------------|----------|
| INCOME | 正 | increaseBalance(walletId, amount) |
| OUTCOME | 负 | decreaseFrozen(walletId, \|amount\|) |
| FEE | 负 | deductBalance(wallet, \|amount\|) |
| FEE_REVENUE | 正 | increaseBalance(platformWalletId, amount) |
| REFUND | 正 | thawBalance(wallet, amount) |

## 四、处理流程（消费 → 落库 → 执行）

1. **消费**：FundInstructionConsumer 消费 **exchange-fund-instruction**，取 record.key() 为 messageId，record.value() 为 payload。
2. **幂等**：若 **processedFundInstructionRepository.existsByMessageIdAndStatus(messageId, PROCESSED)** 为 true，直接 return。
3. **落库**：若该 messageId 尚无记录，则插入 **ProcessedFundInstruction**（messageId, status=PENDING, payload, createdAt）；若已有记录（PENDING/FAILED）则复用，不覆盖 payload。
4. **执行**：解析 payload 为 FundInstructionDTO，对每条 instruction：按 memberId+symbol 查 **MemberWallet**，再按 type 调 MemberWalletService 对应方法（见上表）。整批在同一事务内执行；任一条失败则抛异常、事务回滚，并更新记录 status=FAILED。
5. **成功**：全部执行成功后更新记录 status=PROCESSED、processed_at。
6. **重试**：**RetryFundInstructionJob** 定时（默认 30s）扫描 status 为 PENDING 或 FAILED 的记录，用其 payload 重新调用 processAndExecute，成功则更新为 PROCESSED。

---

## 五、如何保证资金结算结果不丢、可恢复

| 机制 | 说明 |
|------|------|
| **先落库再执行** | 收到消息后先写入 **processed_fund_instruction**（PENDING + payload），再执行钱包操作；执行失败时 payload 已落库，可重试。 |
| **幂等** | 按 messageId 且 status=PROCESSED 判重：已处理过则不再执行，避免重复消费导致重复加减款。 |
| **事务** | 单条 messageId 下整批 instruction 在同一事务内执行，要么全部成功要么全部回滚，避免部分成功部分失败。 |
| **失败状态与重试** | 执行失败时 status 置为 FAILED，**RetryFundInstructionJob** 定时用已落库的 payload 重新执行，不重新消费 Kafka。 |
| **可恢复** | 重启或修复（如补建平台钱包）后，定时任务继续重试 PENDING/FAILED；也可按 messageId 从表查询并人工/脚本重放。 |

## 六、改造过程记录

### 6.1 exchange-core

- **ProcessedFundInstruction** 实体：表 processed_fund_instruction，字段 id, message_id(唯一), status(PENDING/PROCESSED/FAILED), payload, created_at, processed_at。
- **ProcessedFundInstructionRepository**：existsByMessageIdAndStatus；findByMessageId；findByStatusInOrderByIdAsc(List&lt;Status&gt;) 供重试任务使用。

### 6.2 market

- **market** 增加对 **core** 的依赖，以便注入 MemberWalletService。
- **FundInstructionExecuteService**：processAndExecute(messageId, payload) 幂等 → 落库 PENDING（或复用已有记录）→ 解析 payload → 按条执行钱包操作 → 成功则 PROCESSED，失败则 FAILED；retryExecutePending() 扫描 PENDING/FAILED 并重新执行。
- **FundInstructionConsumer**：@KafkaListener(topics = "exchange-fund-instruction")，调用 **fundInstructionExecuteService.processAndExecute(record.key(), record.value())**。
- **RetryFundInstructionJob**：定时（fund.instruction.retry.interval，默认 30000ms）调用 **fundInstructionExecuteService.retryExecutePending()**。

### 6.3 配置项（可选）

- **settlement.platform.memberId**：平台账户 memberId，与结算一致，用于 FEE_REVENUE 指令（默认 **0**）。
- **fund.instruction.retry.interval**：资金指令重试任务间隔（毫秒），默认 **30000**。

---

## 七、涉及模块与类汇总

| 模块 | 类/表 | 说明 |
|------|--------|------|
| exchange-core | ProcessedFundInstruction | 已处理资金指令记录，表 processed_fund_instruction。 |
| exchange-core | ProcessedFundInstructionRepository | JPA 仓库，幂等与重试查询。 |
| core | MemberWalletService | 钱包加减可用、扣减冻结、解冻等。 |
| market | FundInstructionExecuteService | 解析资金指令、落库、执行钱包操作、重试。 |
| market | FundInstructionConsumer | 消费 exchange-fund-instruction。 |
| market | RetryFundInstructionJob | 定时重试未成功执行的资金指令。 |
| Kafka | exchange-fund-instruction | 资金指令来源 topic（结算产出）。 |

## 八、数据示例：一条结算资金指令 → 资金处理结果

以下沿用《用户下单-结算过程》中同一批的结算结果：messageId、symbol、ts 及 instructions 与第八章一致，此处只列与资金处理相关的表与结果。

### 8.1 输入：Kafka exchange-fund-instruction 一条消息

**key**：`msg-a1b2c3d4-e5f6-7890-abcd-ef1234567890`

**value（FundInstructionDTO）**：与结算过程文档中「资金指令 payload」一致，共 8 条 instruction（买方 100：INCOME BTC、OUTCOME USDT、FEE BTC；卖方 200：INCOME USDT、OUTCOME BTC、FEE USDT；平台 0：FEE_REVENUE BTC、FEE_REVENUE USDT）。此处省略完整 JSON，仅列出与「落库 + 改钱包」对应的关系。

### 8.2 落库：processed_fund_instruction 表行记录

| 字段 | 示例值 |
|------|--------|
| id | 1 |
| message_id | msg-a1b2c3d4-e5f6-7890-abcd-ef1234567890 |
| status | PROCESSED |
| payload | （完整 FundInstructionDTO JSON，同上） |
| created_at | 2024-11-14 12:00:03 |
| processed_at | 2024-11-14 12:00:04 |

### 8.3 执行结果：member_wallet 变动（行记录示例）

假设执行前：用户 100 的 BTC 可用=0、USDT 可用=0、USDT 冻结=500；用户 200 的 BTC 冻结=0.01、USDT 可用=0；平台 0 的 BTC/USDT 钱包存在且初始为 0。执行 8 条指令后，变动如下（仅列出涉及变动的字段与变化量，正数为增、负数为减）。

| member_id | coin_id（symbol） | balance 变化 | frozen_balance 变化 | 对应指令 |
|-----------|-------------------|--------------|---------------------|----------|
| 100 | BTC | +0.01, -0.00001 | 0 | INCOME +0.01，FEE -0.00001 → 净 +0.00999 |
| 100 | USDT | 0 | -500 | OUTCOME -500（扣减冻结） |
| 200 | USDT | +500, -0.5 | 0 | INCOME +500，FEE -0.5 → 净 +499.5 |
| 200 | BTC | 0 | -0.01 | OUTCOME -0.01（扣减冻结） |
| 0 | BTC | +0.00001 | 0 | FEE_REVENUE +0.00001 |
| 0 | USDT | +0.5 | 0 | FEE_REVENUE +0.5 |

即：**买方 100** 最终 BTC 可用 +0.00999、USDT 冻结 -500；**卖方 200** 最终 USDT 可用 +499.5、BTC 冻结 -0.01；**平台 0** BTC 可用 +0.00001、USDT 可用 +0.5。

### 8.4 对应关系小结（一条资金指令消息 → 处理结果）

| 输入 | 输出 |
|------|------|
| Kafka exchange-fund-instruction 一条（key=messageId, value=payload） | 1 条 **processed_fund_instruction** 记录（先 PENDING 后 PROCESSED，存 payload） |
| payload 内 8 条 instructions | 对 **member_wallet** 的多次 update：INCOME/OUTCOME/FEE/FEE_REVENUE 分别对应 increaseBalance、decreaseFrozen、deductBalance、increaseBalance；本示例无 REFUND |

资金服务不产出新的 Kafka 消息；对账时可对照 **processed_fund_instruction.payload** 与 **member_wallet** 余额、冻结变化及业务流水。
