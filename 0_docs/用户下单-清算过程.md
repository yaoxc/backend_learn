# 用户下单：清算过程

本文档说明**清算服务**的实现：清算输入与输出、处理流程、落库与 Kafka 的关系，以及如何保证**清算结果不丢、可恢复**；并记录改造过程。

---

## 一、清算在整体链路中的位置

```
订单入口 → 撮合(CoinTrader) → Kafka(exchange-match-result)
    → Market: ExchangeTradeConsumer.handleMatchResult
    → 1) processMatchResultIdempotent（订单/钱包/流水落库）
    → 2) 行情与推送
    → 3) 清算：ClearingService.processAndPublish（计算 → 落库 clearing_result → 发 Kafka exchange-clearing-result）
```

清算在**订单与钱包已落库之后**执行，使用同一批 `trades`、`completedOrders` 做**只读计算**，产出结构化清算结果供下游结算/资金消费。

---

## 二、清算输入

### 2.1 清算数据依据：交易流水与已完成订单（两者都用）

清算结果**同时依据**两类输入计算，缺一不可：

| 输入 | 含义 | 用途 | 产出 |
|------|------|------|------|
| **交易流水（trades）** | 本批撮合产生的每笔成交（ExchangeTrade 列表） | 按**每笔成交**计算买卖双方的手续费、应收应付（与 processOrder 逻辑一致） | **tradeClearingList**：每笔成交对应买卖各一条清算项（fee、incomeSymbol/Amount、outcomeSymbol/Amount） |
| **已完成订单（completedOrders）** | 本批撮合中已完全成交的订单（ExchangeOrder 列表） | 按**每个已完全成交订单**计算应退的冻结金额（与 orderRefund 逻辑一致） | **orderRefundList**：每个订单一条退冻项（orderId、coinSymbol、refundAmount） |

实现上，**ClearingComputeService.compute(messageId, symbol, ts, trades, completedOrders)** 中：遍历 `trades` 调用 `computeTradeItem(order, trade, coin)` 生成 tradeClearingList；遍历 `completedOrders` 调用 `computeRefundItem(order)` 生成 orderRefundList。两类一起组成一条清算结果（ClearingResultDTO）供落库与下发。

### 2.2 输入来源与触发

- **来源**：与订单处理同源，均为 Kafka topic **exchange-match-result** 的单条消息（MatchResult）。
- **内容**：`messageId`、`symbol`、`ts`、`trades`（成交明细列表）、`completedOrders`（已完全成交订单列表）。
- **触发**：独立清算消费者 **ClearingMatchResultConsumer** 订阅 exchange-match-result（group=market-clearing-group），解析后调用 **ClearingService.processAndPublish(messageId, symbol, ts, trades, completedOrders)**；或（未拆分时）在 ExchangeTradeConsumer.handleMatchResult 中订单与推送完成后调用。
- **幂等**：`processAndPublish` 内若 **clearing_result** 表已存在该 `messageId`，直接返回，不重复计算、不重复落库、不重复发 Kafka。

## 三、清算输出

### 3.1 先落库再发 Kafka

| 输出 | 说明 |
|------|------|
| **落库** | 表 **clearing_result**（exchange-core 实体），字段：id, message_id(唯一), symbol, ts, status(PENDING/PUBLISHED/FAILED), payload(JSON 文本), created_at, published_at。 |
| **Kafka** | Topic **exchange-clearing-result**，单条消息 key=messageId，value=payload（与表中 payload 一致）。 |

顺序：**先写入 clearing_result（status=PENDING），再发 Kafka；发送成功则更新 status=PUBLISHED、published_at；发送失败则更新 status=FAILED**，由定时任务重试发送。

### 3.2 payload 结构（ClearingResultDTO）

- **messageId / symbol / ts**：与 MatchResult 一致。
- **tradeClearingList**：每笔成交对应的买卖双方清算项。每项含：orderId, direction(BUY/SELL), fee, incomeSymbol, incomeAmount, outcomeSymbol, outcomeAmount（与 ExchangeOrderService.processOrder 中手续费、收支计算一致）。
- **orderRefundList**：每个已完全成交订单的退冻结项。每项含：orderId, coinSymbol, refundAmount（与 orderRefund 公式一致：frozenBalance - dealBalance）。

下游（结算/资金服务）可订阅 **exchange-clearing-result**，按 payload 生成资金指令，无需再查订单与成交表。

## 四、处理流程（消费 → 计算 → 落库 → 发 Kafka）

1. **消费**：与订单处理共用 **exchange-match-result**，在 handleMatchResult 中订单与钱包处理完成后调用清算。
2. **幂等**：若 `clearingResultRepository.existsByMessageId(messageId)` 为 true，直接 return。
3. **计算**：**ClearingComputeService.compute(messageId, symbol, ts, trades, completedOrders)**：
   - 按交易对取费率（ExchangeCoinService.findBySymbol）；
   - 对每笔成交加载买卖订单，按 processOrder 逻辑计算 fee、income、outcome（机器人/管理员 fee=0）；
   - 对每个已完成订单按 orderRefund 逻辑计算退冻结金额；
   - 返回 ClearingResultDTO，序列化为 JSON 作为 payload。
4. **落库**：构造 **ClearingResult**（messageId, symbol, ts, status=PENDING, payload, createdAt），**clearingResultRepository.save**。
5. **发 Kafka**：**kafkaTemplate.send(clearingResultTopic, messageId, payload)**；成功则更新 status=PUBLISHED、publishedAt 并 save；失败则更新 status=FAILED 并 save，打日志。
6. **重试**：**RetryClearingPublishJob** 定时（默认 30s）扫描 status 为 PENDING 或 FAILED 的记录，用其 payload 重新发 Kafka，成功则更新为 PUBLISHED 和 publishedAt。

## 五、如何保证清算结果不丢、可恢复

| 机制 | 说明 |
|------|------|
| **先落库再发 Kafka** | 任一条清算结果先写入 **clearing_result** 再发送；发送失败时数据库中已有完整 payload，不会因 Kafka 故障丢失结果。 |
| **幂等** | 按 messageId 去重：已存在则不再计算、不重复落库、不重复发送，避免重复消费导致重复下发。 |
| **失败状态与重试** | 发送失败时 status 置为 FAILED（或保持 PENDING），**RetryClearingPublishJob** 定时对 PENDING/FAILED 记录重新发送，不重新计算，仅用已有 payload 重发。 |
| **可恢复** | 重启或 Kafka 恢复后，定时任务会继续重试未发送成功的记录；也可按 messageId 或时间范围从 **clearing_result** 表查询并人工/脚本重放。 |
| **可对账** | 清算结果持久化在 **clearing_result**，可与下游消费位点、结算/资金流水对账。 |

## 六、改造过程记录

### 6.1 exchange-core

- **ClearingResult** 实体：表名 clearing_result，字段 id, message_id(唯一), symbol, ts, status, payload, created_at, published_at。
- **ClearingResultRepository**：existsByMessageId；findByStatusInOrderByIdAsc(List&lt;Status&gt;) 供重试任务使用。
- **ClearingResultDTO**：清算结果 DTO，含 tradeClearingList、orderRefundList，用于 payload 与 Kafka 消息体。
- **ClearingComputeService**：入参 messageId, symbol, ts, trades, completedOrders；只做计算（与 processOrder/orderRefund 逻辑一致），不写钱包与订单；返回 ClearingResultDTO。

### 6.2 market

- **ClearingService**：注入 ClearingResultRepository、ClearingComputeService、KafkaTemplate；实现 processAndPublish（幂等 → 计算 → 落库 PENDING → 发 Kafka → 成功则 PUBLISHED，失败则 FAILED）、retryPublishPending（扫描 PENDING/FAILED 重试发送）。
- **ExchangeTradeConsumer.handleMatchResult**：在订单处理与推送之后调用 **clearingService.processAndPublish(messageId, symbol, ts, trades, completedOrders)**。
- **RetryClearingPublishJob**：定时（clearing.retry.interval，默认 30000ms）调用 **clearingService.retryPublishPending()**。

### 6.3 配置项（可选）

- **clearing.result.topic**：清算结果 Kafka topic，默认 **exchange-clearing-result**。
- **clearing.retry.interval**：重试任务间隔（毫秒），默认 **30000**。

## 七、涉及模块与类汇总

| 模块 | 类/表 | 说明 |
|------|--------|------|
| exchange-core | ClearingResult | 清算结果实体，表 clearing_result。 |
| exchange-core | ClearingResultRepository | JPA 仓库，幂等与重试查询。 |
| exchange-core | ClearingResultDTO / TradeClearingItem / OrderRefundItem | 清算结果 DTO 与 payload 结构。 |
| exchange-core | ClearingComputeService | 清算计算（仅计算，不写库）。 |
| market | ClearingService | 清算流程：计算、落库、发 Kafka、重试。 |
| market | ExchangeTradeConsumer | 消费 exchange-match-result，其中调用 ClearingService.processAndPublish。 |
| market | RetryClearingPublishJob | 定时重试未成功发送的清算结果。 |
| Kafka | exchange-clearing-result | 清算结果下游 topic。 |

## 八、数据示例：一条订单 → 成交流水 → 清算结果

以下为**单笔成交、买卖双方各一单且该笔成交后两单均完全成交**的简化示例。交易对 **BTC/USDT**，手续费率 0.1%（0.001）。

### 8.1 场景与假设

- 买方订单 BO001：限价买入 0.01 BTC，价格 50000 USDT（冻结 500 USDT）。
- 卖方订单 SO001：限价卖出 0.01 BTC，价格 50000 USDT（冻结 0.01 BTC）。
- 撮合产生 1 笔成交：价格 50000，数量 0.01，买卖双方该笔后均完全成交。

| direction | BUY | SELL |

| price | 50000 | 50000 |
| traded_amount | 0.01 | 0.01 |
| turnover | 500 | 500 |
| coin_symbol | BTC | BTC |
| base_symbol | USDT | USDT |
| status | COMPLETED | COMPLETED |

**payload（即 Kafka exchange-clearing-result 的 value）JSON：**

```json
{
  "messageId": "msg-a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "symbol": "BTC/USDT",
  "ts": 1699900000000,
  "tradeClearingList": [
    {
      "memberId": 100,
      "orderId": "BO001",
      "direction": "BUY",
      "fee": 0.00001,
      "incomeSymbol": "BTC",
      "incomeAmount": 0.00999,
      "outcomeSymbol": "USDT",
      "outcomeAmount": 500
    },
    {
      "memberId": 200,
      "orderId": "SO001",
      "direction": "SELL",
      "fee": 0.5,
      "incomeSymbol": "USDT",
      "incomeAmount": 499.5,
      "outcomeSymbol": "BTC",
      "outcomeAmount": 0.01
    }
  ],
  "orderRefundList": []
}
```

- **tradeClearingList**：与本批成交一一对应，每条成交产生买卖两条清算项（手续费、应收币种/金额、应付币种/金额），与 processOrder 逻辑一致。
- **orderRefundList**：本示例中两单该笔成交后均完全成交，退冻结 = 冻结 - 已成交 = 0，故为空数组；若有部分成交后完成一单，会有一条 orderId + coinSymbol + refundAmount（退冻金额）。
