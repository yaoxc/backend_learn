# 下单链路引入“数据中转服务”改造方案（已落地到代码）

## 目标

把 **下单服务（exchange-api）** 与 **撮合服务（exchange）** 的耦合点从“直连同一个 Kafka topic”改为：

- 下单只写 **对外入口 topic（ingress）**
- 数据中转服务（exchange-relay）消费 ingress，做统一治理后转发到 **撮合内部 topic（internal）**
- 撮合仍只消费 internal

这样可以在不侵入撮合核心的情况下获得：
- topic 权限边界更清晰（外部只允许写 ingress）
- 统一分区键策略（按 `symbol`），保证撮合端同交易对顺序更稳定
- 后续可在 relay 做 schema 版本治理、灰度、风控前置、限流、审计留痕等

---

## Topic 设计

### 入口 topic（对外）
- `exchange-order-ingress`：下单入口
- `exchange-order-cancel-ingress`：撤单入口

### 内部 topic（撮合）
- `exchange-order`：撮合消费的下单 topic（保持原样）
- `exchange-order-cancel`：撮合消费的撤单 topic（保持原样）

---

## 拓扑关系：谁连 ingress，谁连 internal

一句话概括现在的设计：

> **所有生产者统一发到 `*-ingress`，所有“核心内部消费者（撮合/清算等）统一只连 internal topic，中间由 `exchange-relay` 做桥接与治理。**

### 生产端（Producer）——只写 ingress topic

- `exchange-api`（下单/撤单）：
  - 下单：`exchange-order-ingress`
  - 撤单：`exchange-order-cancel-ingress`
- 未来新增的下单入口（如行情机器人、外部撮合接入）：
  - 建议也只写上述 ingress topic，而不是直接写 `exchange-order` / `exchange-order-cancel`。

**意义**：所有来自“外部世界”的订单请求都先进入 ingress topic，权限控制、限流、审计等都可以在入口或 relay 层统一做。

### 数据中转服务（exchange-relay）——“入口 → 内部”的桥接层

- 消费：
  - `exchange-order-ingress`
  - `exchange-order-cancel-ingress`
- 转发：
  - 到内部下单 topic：`exchange-order`
  - 到内部撤单 topic：`exchange-order-cancel`
- 转发策略：
  - 以 `symbol` 作为 message key 转发，保证同一交易对的订单更大概率落到同一分区，撮合端单分区顺序消费更稳定。

你可以把 relay 理解为：

> **「生产者世界」与「撮合内部世界」之间的一道“数据闸门 + 协议适配器」**。

### 撮合/内部消费端（Consumer）——只连 internal topic

- 撮合服务 `exchange`（`ExchangeOrderConsumer`）：
  - 只监听 `exchange-order`，收到订单后路由到 `CoinTrader.trade(order)`。
  - 只监听 `exchange-order-cancel`，收到撤单请求后调用 `CoinTrader.cancelOrder(order)`。
- 如果未来有内部服务需要直接消费“原始订单请求”（而非成交/结果），推荐也连 internal topic，而不是直接连 ingress topic。

**效果总结**：
- 下单服务的视角：只是从“写 `exchange-order`”变成了“写 `exchange-order-ingress`”，业务代码变化极小。
- 撮合服务的视角：完全无感知，不关心 ingress，只认 `exchange-order` / `exchange-order-cancel`。
- 运维/架构视角：通过 ingress + relay 把“接入治理”从撮合核心中抽离出来，后续任何协议变更、风控/限流、灰度发布都可以在这一层演进。

## 代码改动点

### 1）新增服务模块：`exchange-relay`

模块：`exchange-relay/`

核心类：`com.bizzan.bitrade.consumer.ExchangeOrderRelayConsumer`

行为：
- 消费 ingress topic
- 解析 `ExchangeOrder` JSON
- 以 `symbol` 作为 key 转发到 internal topic
- **payload 保持不变**（仍是 `ExchangeOrder` 的 JSON），尽量不改撮合消费者

配置（默认值）：
- `relay.order.ingress-topic=exchange-order-ingress`
- `relay.order.cancel-ingress-topic=exchange-order-cancel-ingress`
- `relay.order.internal-topic=exchange-order`
- `relay.order.internal-cancel-topic=exchange-order-cancel`

---

### 2）下单服务改造：`exchange-api`

文件：`exchange-api/src/main/java/com/bizzan/bitrade/controller/OrderController.java`

改动：
- 原本：落单后直接 `kafkaTemplate.send("exchange-order", ...)`。
- 现在：
  - **先写 ingress topic**：仍发往 `orderIngressTopic`（默认 `exchange-order-ingress`）、撤单发往 `cancelIngressTopic`（默认 `exchange-order-cancel-ingress`）。
  - **本地消息表**：下单改为通过 **OrderOutboxService.addOrderAndSaveOutbox**（落单 + 写 outbox 同一事务），再 **trySendById** 发 Kafka；撤单改为 **saveCancelOutbox** + **trySendById**。发送失败由 **RetryOrderOutboxJob** 定时重试，保证最终投递成功。详见下节「3）下单服务本地消息表改造」。

可通过配置覆盖：
- `exchange.relay.order.ingress-topic`
- `exchange.relay.order.cancel-ingress-topic`

---

### 3）下单服务本地消息表改造：保证 Kafka 投递成功（已实现）

为规避「订单已落库但 Kafka 发送失败」导致撮合侧收不到订单，采用**本地消息表**：与下单/撤单同库同事务写入 outbox，再异步发 Kafka，发送失败由定时任务重试。

#### 3.1 表与组件

| 项目 | 说明 |
|------|------|
| **表** | **order_kafka_outbox**（exchange-core 实体 `OrderKafkaOutbox`）：id, topic, message_key, payload, status(PENDING/SENT/FAILED), created_at, sent_at。与业务同库。 |
| **下单** | **OrderOutboxService.addOrderAndSaveOutbox(memberId, order, orderIngressTopic)**：同一事务内先 `orderService.addOrder()` 再插入一条 outbox（status=PENDING），返回 outbox id；Controller 随后 **trySendById(outboxId)** 立即尝试发送，失败不阻塞接口。 |
| **撤单** | **OrderOutboxService.saveCancelOutbox(cancelIngressTopic, orderId, payload)**：插入一条 outbox（PENDING），再 **trySendById(outboxId)**。 |
| **发送** | **trySendById(outboxId)**：按 id 取 outbox，发 Kafka，成功则 status=SENT、sent_at 更新，失败则 status=FAILED。 |
| **重试** | **RetryOrderOutboxJob**（定时，默认 30s）：查询 status 为 PENDING/FAILED 的记录，逐个 trySendById，保证最终投递成功。 |

#### 3.2 流程简述

1. **用户下单**：Controller 调用 `orderOutboxService.addOrderAndSaveOutbox(...)` → 同一事务内落单 + 写一条 outbox（topic=orderIngressTopic, message_key=orderId, payload=订单 JSON）→ 返回后立即 `trySendById(outboxId)`；失败则由定时任务重试。
2. **用户撤单**：Controller 调用 `orderOutboxService.saveCancelOutbox(...)` 写 outbox → `trySendById(outboxId)`；失败由定时任务重试。
3. **定时任务**：`RetryOrderOutboxJob` 按 **exchange.outbox.retry.interval**（默认 30000 ms）扫描 PENDING/FAILED，发送并更新状态。

#### 3.3 涉及模块与配置

| 模块 | 类/表 | 说明 |
|------|--------|------|
| exchange-core | OrderKafkaOutbox | 实体，表 order_kafka_outbox。 |
| exchange-core | OrderKafkaOutboxRepository | JPA 仓库，findByStatusInOrderByIdAsc。 |
| exchange-api | OrderOutboxService | addOrderAndSaveOutbox、saveCancelOutbox、trySendById、retryPending。 |
| exchange-api | RetryOrderOutboxJob | 定时重试未发送成功的 outbox。 |
| exchange-api | OrderController | 下单/撤单改为调用 OrderOutboxService，不再直接 kafkaTemplate.send。 |

**配置项**：

- **exchange.outbox.retry.interval**：重试间隔（毫秒），默认 **30000**。

**建表**（若未使用 JPA 自动建表）：

```sql
CREATE TABLE order_kafka_outbox (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  topic VARCHAR(128) NOT NULL,
  message_key VARCHAR(64) NOT NULL,
  payload TEXT NOT NULL,
  status VARCHAR(20) NOT NULL,
  created_at DATETIME NOT NULL,
  sent_at DATETIME NULL,
  INDEX idx_outbox_status (status),
  INDEX idx_outbox_topic_key (topic, message_key)
);
```

## 运行/验证方式（最小路径）

1. 启动 Kafka、Eureka（按项目既有 docker-compose 或部署方式）
2. 启动撮合服务 `exchange`（仍监听 internal topic：`exchange-order`）
3. 启动中转服务 `exchange-relay`（监听 ingress 并转发到 internal）
4. 启动下单服务 `exchange-api`（下单写 ingress）
5. 调用下单接口 `/order/add`，观察：
   - `exchange-api` 往 `exchange-order-ingress` 发消息
   - `exchange-relay` 消费并转发到 `exchange-order`（key = symbol）
   - `exchange` 消费 `exchange-order` 并进入 `CoinTrader.trade(order)`
