# 架构分析：按需扩容、消息不丢、服务堵塞隔离

本文档基于**当前实现**，分析「订单服务、撮合服务、清算服务、结算服务、资金服务、market」在以下方面的情况：

1. **是否可以根据需要自由扩容**
2. **消息是否不丢失**
3. **宕机重启是否可恢复**
4. **任一服务出现堵塞，是否不会影响其它服务**

---

## 一、当前架构与部署形态

### 1.1 数据流与模块归属

```
订单(API) → 撮合(exchange) → Kafka: exchange-match-result
                                    │
        ┌───────────────────────────┼───────────────────────────┐
        │                           │                             │
        ▼                           ▼                             ▼
ExchangeTradeConsumer        ClearingMatchResultConsumer    (其它 group 可选)
(订单状态+行情+推送)                 │
默认 group                          ▼
                            clearing_result 落库
                                    │
                                    ▼
                            Kafka: exchange-clearing-result
                                    │
                                    ▼
                            SettlementConsumer → settlement_result 落库
                                    │
                                    ▼
                            Kafka: exchange-fund-instruction
                                    │
                                    ▼
                            FundInstructionConsumer → 改 member_wallet
```

- **撮合**：在 **exchange** 模块；通过 `QueueAndWalMatchResultPublisher` 写 WAL + 后台发 `exchange-match-result`，**不等待**下游消费。
- **订单状态/行情/推送、清算、结算、资金**：当前均在 **market** 模块内实现：
  - `ExchangeTradeConsumer`：消费 `exchange-match-result`（默认 group）
  - `ClearingMatchResultConsumer`：消费 `exchange-match-result`（**group=market-clearing-group**）
  - `SettlementConsumer`：消费 `exchange-clearing-result`（默认 group）
  - `FundInstructionConsumer`：消费 `exchange-fund-instruction`（默认 group）

即：**清算、结算、资金、market（订单状态+行情+推送）在代码上通过不同 Kafka topic/group 解耦，但部署上同属一个 market 应用进程**。订单服务（接收下单请求、写入订单并驱动撮合）可能在 exchange 或 exchange-api，与撮合同机或同进程，需看实际部署。

## 二、各问题的逐项分析

### 2.1 是否可以根据需要自由扩容？

| 维度 | 结论 | 说明 |
|------|------|------|
| **撮合 (exchange)** | ✅ 可独立扩容 | 单独部署；多实例时需按 symbol 分片或约定同一 symbol 只在一处撮合，否则会重复撮合。按 symbol 分片后，可对热点 symbol 单独加实例。 |
| **消费端 (market 应用)** | ⚠️ 可扩容，但非「按服务」自由扩 | 多起几个 market 实例即可拉高消费能力：<br>• **exchange-match-result** 有两个 group（默认 + market-clearing-group），所以「订单+行情+推送」与「清算」可**各自**通过增加该 group 的 consumer 数扩容（同一 group 内分区分配）。<br>• **exchange-clearing-result / exchange-fund-instruction** 只有默认 group，扩 market 实例 = 同时扩结算与资金消费。<br>• **当前不能**「只扩清算不扩行情」或「只扩资金不扩结算」，因为都在同一应用里；若要真正按服务自由扩，需把清算、结算、资金拆成**独立部署的应用**（各自独立 group、独立实例数）。 |
| **订单服务** | 视部署而定 | 若订单与撮合在同一应用，扩实例需配合 symbol 分片；若订单在独立 API 服务，则可单独水平扩展。 |

**小结**：  
- 撮合可独立扩；market 整体可扩，且 match-result 上「订单+行情」与「清算」已分 group，可分别加 consumer。  
- 「按需自由扩」在**逻辑上**通过 Kafka 与不同 group 已支持，但**部署上**清算/结算/资金仍在同一 market 进程，若要「只扩清算」或「只扩资金」需拆成独立服务并单独部署。

### 2.2 消息是否不丢失？

| 环节 | 机制 | 结论 |
|------|------|------|
| **撮合 → Kafka** | 内存队列 + **本地 WAL**；Writer 顺序写 WAL，Sender 从 WAL 读并发 Kafka，**成功后推进 offset**；宕机重启从未发位置继续；队列满时撮合线程 **put 阻塞**，不丢条。 | ✅ 在 WAL 与队列未损、磁盘可靠前提下，**可做到不丢**；若希望更稳可对 WAL 做 fsync/多副本（当前为单机 WAL）。 |
| **Kafka 自身** | 依赖 Kafka 的 replication、ack、retention；生产通常 ack=all、min.insync.replicas≥1、retention 足够长。 | ✅ 集群配置合理则消息不丢。 |
| **match-result → 订单+行情** | 先幂等落库（processed_match_result_message），再推送/行情；重复消费只导致幂等跳过。 | ✅ 不丢（依赖 Kafka 不丢 + 消费至少一次）。 |
| **match-result → 清算** | ClearingMatchResultConsumer 消费后先落库 clearing_result（PENDING），再发 Kafka；发失败标 FAILED，**RetryClearingPublishJob** 定时重试。 | ✅ 先落库再发，失败可重试，**不丢**。 |
| **clearing-result → 结算** | 先落库 settlement_result（PENDING），再发 Kafka；失败标 FAILED，**RetrySettlementPublishJob** 重试。 | ✅ **不丢**。 |
| **fund-instruction → 资金** | 先落库 processed_fund_instruction（PENDING），再执行钱包；失败标 FAILED，**RetryFundInstructionJob** 重试。 | ✅ **不丢**。 |

**小结**：  
在当前实现下（WAL + 先落库再发/再执行 + 幂等 + 定时重试），**消息不丢**在设计与实现上已具备；实际是否不丢还依赖：Kafka 集群配置、磁盘与 WAL 可靠性、重试任务正常运行。

### 2.3 宕机重启是否可恢复？

| 环节 | 恢复方式 | 结论 |
|------|----------|------|
| **撮合 (exchange)** | 重启后 Sender 从 **WAL offset 文件**读取上次发到的位置，从 WAL 该位置继续读、继续发 Kafka；未发出去的记录都在 WAL 中。 | ✅ 可恢复 |
| **各 Kafka 消费者** | 按 consumer group 提交的 offset 消费；重启后从上次提交的 offset 继续拉取，不会漏消息（在 Kafka retention 内）。 | ✅ 可恢复 |
| **清算** | 已落库为 PENDING/FAILED 的 clearing_result 由 **RetryClearingPublishJob** 定时重试发 Kafka；应用重启后任务照常执行。 | ✅ 可恢复 |
| **结算** | 已落库为 PENDING/FAILED 的 settlement_result 由 **RetrySettlementPublishJob** 重试发 Kafka；重启后照常。 | ✅ 可恢复 |
| **资金** | 已落库为 PENDING/FAILED 的 processed_fund_instruction 由 **RetryFundInstructionJob** 重试执行钱包；重启后照常。 | ✅ 可恢复 |

**小结**：  
宕机重启后，撮合靠 WAL + offset 文件恢复未发消息；下游消费靠 Kafka offset；清算/结算/资金靠「已落库 + 定时重试任务」恢复未发或未执行完的数据，**整体可恢复**。

### 2.4 任一服务堵塞，是否不会影响其它服务？

| 场景 | 影响分析 | 结论 |
|------|----------|------|
| **撮合堵塞** | 撮合**不等待**下游；只受本机队列与 WAL 影响：若 Writer/Sender 慢或卡住，队列会满，撮合线程在 **queue.put()** 上阻塞，形成背压。下游（market、清算等）消费再慢也**不会反向阻塞撮合**（已通过 Kafka 解耦）。 | ✅ 下游堵塞**不会**导致撮合堵塞；✅ 撮合自身队列/磁盘慢会阻塞撮合，不影响下游消费。 |
| **Market（订单+行情+推送）消费慢** | exchange-match-result 上「默认 group」lag 增大；**ClearingMatchResultConsumer** 是另一 group，独立拉取，**不受默认 group 进度影响**。若 market 与清算在同一进程，共享 JVM/DB 连接池/CPU，market 消费堵会导致 DB 或 CPU 紧张，**可能间接拖慢同进程内的清算**。 | ⚠️ Kafka 层面**隔离**（不同 group）；**同进程**内资源竞争会互相影响。 |
| **清算消费慢** | 同上，清算用 market-clearing-group；与「订单+行情」消费进度独立。同进程内若清算占满 DB 或线程，可能拖慢 SettlementConsumer、FundInstructionConsumer、ExchangeTradeConsumer。 | ⚠️ 同上。 |
| **结算或资金消费慢** | exchange-clearing-result / exchange-fund-instruction 各自 topic，互不阻塞；但结算与资金都在 market 应用内，**共享进程资源**，一个堵可能拖慢另一个。 | ⚠️ 不同 topic 互不阻塞；同进程内会互相影响。 |
| **订单服务堵塞** | 若订单与撮合同进程，订单接口慢或 DB 慢可能占用连接/线程，影响撮合；若订单独立部署，则只影响下单延迟，不影响撮合与后续链路。 | 视部署：同进程则可能影响撮合；独立部署则隔离更好。 |

**小结**：  
- **撮合与下游**：通过 Kafka 解耦，**下游堵塞不会导致撮合堵塞**；撮合堵塞只与自身队列/WAL/磁盘有关。  
- **各 Kafka 消费者**：不同 topic、不同 group 的消费进度**互不影响**（如 match-result 上两 group、clearing-result、fund-instruction 彼此独立）。  
- **当前限制**：清算、结算、资金、market 消费者都在**同一 market 进程**内，共享线程池、DB 连接池、内存与 GC；**任一环节严重堵塞（如 DB 慢、死锁、OOM）可能拖垮整个进程**，从而影响同进程内其它消费者。要真正做到「任一服务堵塞不影响其它服务」，需要把清算、结算、资金拆成**独立部署的应用**（独立进程/容器），并做好资源隔离。

## 三、消息不丢失与宕机重启可恢复：前提与注意点

在满足下列前提时，当前架构可做到**消息不丢失、宕机重启可恢复**：

| 前提 | 说明 |
|------|------|
| **WAL 与 offset 文件** | 撮合侧 WAL 与 offset 文件在 exchange 本机磁盘；若磁盘损坏或 WAL 丢失，撮合未发出去的那段无法恢复。生产建议：WAL 目录放在可靠存储上，必要时做备份或多机方案。 |
| **Kafka** | retention 足够长，避免宕机或 lag 期间未消费的 partition 被清理；ack、replication、min.insync.replicas 按生产要求配置。 |
| **重试任务** | RetryClearingPublishJob、RetrySettlementPublishJob、RetryFundInstructionJob 随应用启动且调度周期合理（如默认 30s），否则「先落库再发/再执行」的失败分支无法自动恢复。 |
| **幂等** | 下游按 messageId（及 processed_match_result_message、clearing_result、settlement_result、processed_fund_instruction）做幂等，重复消费或重试不会导致重复加款/重复落库。 |

**结论**：在 WAL/磁盘可靠、Kafka 配置合理、重试任务常驻的前提下，**可以做到消息不丢失、宕机重启可恢复**。

## 四、总结表

| 目标 | 当前架构下的结论 | 若要完全达成的建议 |
|------|------------------|---------------------|
| **1. 按需自由扩容** | 撮合可独立扩；market 整体可扩，且 match-result 上「订单+行情」与「清算」已分 group 可分别加 consumer；结算/资金与 market 同进程，不能单独「只扩清算」或「只扩资金」。 | 将清算、结算、资金拆成**独立服务**（独立应用/镜像），各自订阅对应 topic、独立扩缩容。 |
| **2. 消息不丢失** | 已具备：撮合侧 WAL + 先落库再发/再执行 + 幂等 + 定时重试；依赖 Kafka 与磁盘配置合理。 | 保持现有设计；生产环境保证 Kafka ack、replication、retention 与 WAL 磁盘可靠；监控重试任务与 lag。 |
| **2'. 宕机重启可恢复** | 已具备：撮合靠 WAL+offset；消费者靠 Kafka offset；清算/结算/资金靠已落库 + 定时重试任务。 | 见第三节：WAL/磁盘可靠、Kafka retention、重试任务常驻、幂等。 |
| **4. 任一服务堵塞不影响其它** | 撮合与下游已解耦；不同 topic/group 的 Kafka 消费互不影响。**但**清算/结算/资金/market 同进程，资源竞争会导致「一个堵、整进程慢」。 | 将清算、结算、资金（及必要时推送）拆成**独立部署**，实现进程级隔离；再配合资源限制（CPU/内存/连接池）与熔断，使单服务堵塞不拖垮其它服务。 |

## 五、改进方案：清算、结算、资金与 market 拆分

将**清算、结算、资金与 market** 拆成**独立部署的应用**后，可同时满足上述四点（按需自由扩容、消息不丢失、宕机重启可恢复、任一服务堵塞不影响其它）。

### 5.1 拆分前后对照

| 目标 | 拆分前 | 拆分后 |
|------|--------|--------|
| **1. 按需自由扩容** | 只能整体扩 market，无法单独扩清算/结算/资金 | ✅ 各服务独立部署、独立实例数，可单独扩清算、只扩资金等 |
| **2. 消息不丢失** | ✅ 已具备（WAL + 先落库再发/再执行 + 幂等 + 重试） | ✅ 不变，仍依赖现有设计 + Kafka/磁盘/重试 |
| **3. 宕机重启可恢复** | ✅ 已具备（WAL/offset + 重试任务） | ✅ 不变，各服务各自重启、各自重试即可 |
| **4. 任一服务堵塞不影响其它** | ❌ 同进程共享资源，一个堵会拖慢整进程 | ✅ 进程隔离，清算堵只影响清算，不影响 market/结算/资金 |

### 5.2 拆分形态（四个独立应用）

| 应用 | 职责 | 消费 topic / group | 产出 |
|------|------|-------------------|------|
| **market** | 订单状态 + 行情 + 推送 | `exchange-match-result`（默认 group） | 订单明细/状态、K 线/盘口、WebSocket/Netty 推送 |
| **清算** | 计算清算结果并下发 | `exchange-match-result`（group=market-clearing-group） | clearing_result 落库、发 `exchange-clearing-result` |
| **结算** | 根据清算结果生成资金指令 | `exchange-clearing-result`（独立 group） | settlement_result 落库、发 `exchange-fund-instruction` |
| **资金** | 执行资金指令改钱包 | `exchange-fund-instruction`（独立 group） | processed_fund_instruction 落库、改 member_wallet |

每个应用单独部署（独立进程/容器/镜像），按需配置实例数与资源（CPU、内存、连接池等）；结算、资金需为各自 consumer 配置**独立的 group**，与当前 market 内默认 group 区分，避免与未拆分前的 consumer 重复消费或冲突。

### 5.3 实施说明

拆分形态可以很简单：

- **market**：只跑订单状态 + 行情 + 推送（只消费 `exchange-match-result` 默认 group，或单独一个 group）。
- **清算**：独立应用，只跑 ClearingMatchResultConsumer（`exchange-match-result`，group=market-clearing-group），写 clearing_result、发 `exchange-clearing-result`。
- **结算**：独立应用，只跑 SettlementConsumer（`exchange-clearing-result`），写 settlement_result、发 `exchange-fund-instruction`。
- **资金**：独立应用，只跑 FundInstructionConsumer（`exchange-fund-instruction`），写 processed_fund_instruction、改 member_wallet。

实施时注意：

- **Topic 与 group**：match-result 上「订单+行情」与「清算」已分 group；clearing-result、fund-instruction 在拆分时为结算服务、资金服务配置各自 group 即可。**无需改 Kafka topic 或消息格式**。
- **代码层面**：可将现有 market 模块中的 ClearingMatchResultConsumer + ClearingService、SettlementConsumer + SettlementService、FundInstructionConsumer + FundInstructionExecuteService 分别迁到**独立应用**（新模块或新镜像），仅保留 ExchangeTradeConsumer 与行情/推送在 market 应用；或通过配置/ profile 控制各应用只启动对应消费者。
- **数据库**：clearing_result、settlement_result、processed_fund_instruction、member_wallet 等可继续共用同一库，或按需拆库；各应用需能访问其读写所需的表。
- **效果**：拆分后，消息不丢与宕机可恢复仍依赖现有机制（WAL、先落库再发/再执行、重试任务、幂等）；**按需扩容**与**堵塞隔离**通过进程级隔离实现。
