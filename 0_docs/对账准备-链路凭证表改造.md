## 对账准备：链路凭证表改造记录（match-result / fund-instruction / execution）

### 1) 改造目标

在你当前链路（撮合 `exchange-match-result` → 清算 `exchange-clearing-result` → 结算 `exchange-fund-instruction` → 资金执行落账）中，为了让对账“可证明、可定位”，需要补齐三类**链路凭证表**：

1. **撮合结果凭证**：保存原始 `exchange-match-result`（`messageId` 作为幂等键）
2. **资金指令凭证**：保存结算层产出的资金指令消息（同一 `messageId` 对应一批指令）
3. **资金指令执行结果凭证**：保存资金层执行结果（成功/失败/重试次数/错误信息）

这些表用于：
- 不依赖 Kafka 保留期也能追溯历史（对账/审计）
- 任意差异能从 `messageId` 钻取到“哪一层缺/重/错”
- 配合现有 `member_wallet`（余额事实）与 `member_transaction`（流水凭证）实现闭环对账

---

### 2) 新增表与字段（JPA 自动建表，表名固定）

#### 2.1 `match_result_event`（撮合结果凭证）
- **幂等键**：`message_id` 唯一
- **字段**：
  - `message_id`、`symbol`、`ts`
  - `trades_count`、`completed_orders_count`
  - `payload`（原始 JSON）
  - `created_at`

对应实体：`exchange-core/src/main/java/com/bizzan/bitrade/entity/MatchResultEvent.java`

#### 2.2 `fund_instruction_record`（资金指令凭证）
- **幂等键**：`message_id` 唯一（与 clearingMessageId 对齐）
- **字段**：
  - `message_id`、`symbol`、`ts`
  - `status`：`PENDING / PUBLISHED / FAILED`
  - `payload`（FundInstructionDTO 原始 JSON）
  - `created_at`、`published_at`

对应实体：`exchange-core/src/main/java/com/bizzan/bitrade/entity/FundInstructionRecord.java`

#### 2.3 `fund_instruction_execution`（资金执行结果凭证）
- **幂等键**：`message_id` 唯一（与资金指令消息 key 对齐）
- **字段**：
  - `message_id`
  - `status`：`PENDING / PROCESSED / FAILED`
  - `payload`（资金指令 payload，便于追溯）
  - `error_msg`
  - `retry_count`
  - `created_at`、`processed_at`

对应实体：`exchange-core/src/main/java/com/bizzan/bitrade/entity/FundInstructionExecution.java`

---

### 3) 落库点（代码改造位置）

#### 3.1 清算：消费 `exchange-match-result` 时落 `match_result_event`

文件：
- `clearing/src/main/java/com/bizzan/bitrade/consumer/ClearingMatchResultConsumer.java`

逻辑：
- 解析 `messageId/symbol/ts/trades/completedOrders`
- 先按 `messageId` 幂等写入 `match_result_event`（保存原始 payload + 数量统计）
- 再调用 `ClearingService.processAndPublish(...)` 执行清算

#### 3.2 结算：生成资金指令并发布 Kafka 时落 `fund_instruction_record`

文件：
- `settlement/src/main/java/com/bizzan/bitrade/service/SettlementService.java`

逻辑：
- 生成 `FundInstructionDTO` 的 JSON payload
- 先按 `messageId` 幂等写入 `fund_instruction_record`（PENDING）
- Kafka 发送成功 → 更新为 PUBLISHED + `published_at`
- 发送失败 → 更新为 FAILED（仍可被重试任务补发）

#### 3.3 资金：执行资金指令时落 `fund_instruction_execution`

文件：
- `fund/src/main/java/com/bizzan/bitrade/service/FundInstructionExecuteService.java`

逻辑：
- 保持原有幂等表 `processed_fund_instruction`（PENDING/PROCESSED/FAILED）
- 新增执行结果表：
  - 首次见到 `messageId` → 插入 `fund_instruction_execution`（PENDING + payload）
  - 每次进入执行（含重试）→ `retry_count + 1`
  - 执行成功 → PROCESSED + `processed_at`
  - 执行失败 → FAILED + `error_msg`

---

### 4) 对账如何用这三张表（最小闭环）

以 `messageId` 为主线：

1. `match_result_event`：证明撮合确实产生了该笔结果（trades/订单完成数量、原始 payload）
2. `clearing_result`：证明清算是否计算并发布
3. `settlement_result` + `fund_instruction_record`：证明结算是否生成并发布了资金指令（应执行集合）
4. `processed_fund_instruction` + `fund_instruction_execution`：证明资金是否最终执行成功（以及失败原因/重试次数）
5. `member_wallet` + `member_transaction`：对每个用户币种做余额闭环（期初 + 流水净变动 = 期末）

差异定位示例：
- `match_result_event` 有，但 `clearing_result` 没有 → 清算消费/落库异常
- `fund_instruction_record` 是 FAILED → 结算发 Kafka 失败（可由重试 job 补发）
- `fund_instruction_execution` 是 FAILED 且 retry_count 很高 → 资金执行持续失败，需要排查钱包/余额/冻结不足等

---

### 5) 注意事项（上线/对账实践）

- 这些表的 `payload` 体积可能较大：生产环境建议配合 topic 保留/归档策略与库表分区/清理策略。
- `messageId` 必须全链路稳定：撮合结果、清算、结算、资金执行都应使用同一条链路键做关联。
