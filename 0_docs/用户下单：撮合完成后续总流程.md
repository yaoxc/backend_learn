# 撮合完成后续总流程

本文档说明：撮合引擎将结果发往 Kafka topic **exchange-match-result** 之后，需要完成哪些业务事项；并分别从**当前实现**与**按「市场 + 清算 + 结算 + 资金 + WebSocket」分层**两种视角，说明各服务职责、执行顺序、涉及的数据表与操作。

---

## 一、撮合后发 Kafka 到 exchange-match-result 需要做的事情（业务清单）

从业务角度，一条 MatchResult（一批成交 + 一批已完全成交订单）被发出后，下游需要完成：

| 序号 | 事项 | 说明 |
|------|------|------|
| 1 | **成交明细落库** | 每笔成交写入成交明细表，便于查询、对账、统计。 |
| 2 | **手续费计算与归属** | 按规则计算买卖双方手续费，并可能写入手续费聚合/统计（如按币种、按用户）。 |
| 3 | **钱包变动** | 买方：扣减冻结（计价币）、增加可用（交易币），扣手续费；卖方：扣减冻结（交易币）、增加可用（计价币），扣手续费。 |
| 4 | **资金流水落库** | 每笔资金变动记入流水表，便于审计与对账。 |
| 5 | **推广/返佣** | 若有推荐人，按规则写推荐奖励、流水。 |
| 6 | **订单状态与退冻结** | 已完全成交订单：状态改为 COMPLETED，剩余未成交部分解冻退回。 |
| 7 | **行情与盘口** | 用本批成交更新 K 线、24h 统计、盘口等，供行情接口与盘口推送。 |
| 8 | **实时推送** | 向买卖双方推送「订单部分成交」「订单完全成交」等，供前端/客户端展示。 |

以上 1～6 通常归属「清算 + 结算 + 资金」或合并实现；7 归属行情/市场服务；8 归属推送（WebSocket/Netty 等）服务。

## 二、当前实现：谁消费、做了哪些事、变更了哪些表

当前 **两条独立消费** `exchange-match-result`（不同 consumer group，同一条消息会被两边各消费一次）：

1. **ExchangeTradeConsumer**（market 默认 group）：**handleMatchResult** 做订单状态与成交明细、行情、盘口、推送；**不触发清算、不碰资金**（在 **match.result.use.fund.pipeline=true** 时）。
2. **ClearingMatchResultConsumer**（group=market-clearing-group）：独立消费同一 topic，解析后调用 **clearingService.processAndPublish**，产出清算结果供结算→资金服务使用。

配置 **match.result.use.fund.pipeline**：**true**（默认）= market 仅落订单状态与明细，资金由清算→结算→资金服务执行；**false** = 兼容回退，market 沿用原逻辑直接改钱包。详见 **《用户下单-market消费match-result流程调整说明》**。

### 2.1 消费与幂等

- 消费 topic：`exchange-match-result`（market 与清算各用不同 group 独立消费）。
- **Market 侧**：解析 messageId、trades、completedOrders；**useFundPipeline=true** 时调用 **processMatchResultIdempotentOrderOnly**，**false** 时调用 **processMatchResultIdempotent**。若 messageId 已存在于 **processed_match_result_message**，则跳过整条（不落库、不推送）。
- **清算侧**：ClearingMatchResultConsumer 解析同一条消息，若 trades 与 completedOrders 均为空则跳过，否则调用 clearingService.processAndPublish（清算侧按自身逻辑幂等，如按 messageId 或清算主键）。

### 2.2 落库与资金（exchange-core / 清算，按配置与职责拆分）

**Market 路径（useFundPipeline=true，推荐）**：在 **processMatchResultOrderOnly** 内仅做订单域——

- **每笔成交 processExchangeTradeOrderOnly → processOrderOrderOnly**：写 **exchange_order_detail**、写 **order_detail_aggregation**（MongoDB）；**不**写 member_wallet、member_transaction、不返佣。
- **每个已完全成交订单 tradeCompletedOrderOnly**：更新 **exchange_order**（status=COMPLETED、tradedAmount、turnover、completedTime）；**不**执行 orderRefund（退冻结由资金服务按 REFUND 指令执行）。

**资金**：由清算→结算→资金服务执行（member_wallet、member_transaction、退冻结等）。

**回退路径（useFundPipeline=false）**：在 **processMatchResult** 内与旧逻辑一致——

- **每笔成交 processExchangeTrade → processOrder**：写 exchange_order_detail、order_detail_aggregation；读 member_wallet，increaseBalance / decreaseFrozen；写 member_transaction、返佣。
- **每个已完全成交订单 tradeCompleted**：更新 exchange_order；**orderRefund** 退冻结。

### 2.3 行情与推送（market 模块，在落库之后）

- **CoinProcessor.process(trades)**：用本批成交更新内存 K 线、CoinThumb（24h 等），并经由 **MarketHandler** 写 **K 线存储**（如 MongoDB 或 Redis，依项目配置）。
- **pushJob.addTrades(symbol, trades)**：将本批成交加入推送队列，供盘口/最新成交等推送。
- **messagingTemplate.convertAndSend**（WebSocket）：  
  - 每笔成交推送给买卖双方：`/topic/market/order-trade/{symbol}/{memberId}`（订单部分成交）。  
  - 每个已完全成交订单：`/topic/market/order-completed/{symbol}/{memberId}`（订单完成）。
- **nettyHandler.handleOrder**：与 WebSocket 并行，向 Netty 客户端推送 **PUSH_EXCHANGE_ORDER_TRADE**、**PUSH_EXCHANGE_ORDER_COMPLETED**。

### 2.4 当前实现涉及的数据表汇总

| 数据表/存储 | 操作 | 说明 |
|-------------|------|------|
| processed_match_result_message | INSERT | 幂等：market 记录已处理的 messageId。 |
| exchange_order_detail | INSERT | market：每笔成交一条明细（OrderOnly 路径）。 |
| order_detail_aggregation | INSERT | market：手续费等聚合（MongoDB）。 |
| exchange_order | UPDATE | market：订单状态、tradedAmount、turnover、completedTime。 |
| K 线 / CoinThumb 等 | 更新 | market：CoinProcessor / MarketHandler 写入。 |
| 清算结果（项目内表或 Kafka） | 写/发 | 清算（ClearingMatchResultConsumer）产出，供结算→资金消费。 |
| member_wallet | UPDATE | **资金服务**执行指令：增加可用、扣减冻结、解冻（退冻结）；useFundPipeline=false 时由 market 直接改。 |
| member_transaction | INSERT | **资金服务**执行时写流水（及返佣等）；useFundPipeline=false 时由 market 写。 |
| reward_record | INSERT | useFundPipeline=false 时 market 写；true 时由资金侧或单独任务处理。 |

## 三、若拆成：market、清算、结算、资金、WebSocket 五类服务

下面按**典型 CEX 分层 + 行情 + 推送**，给出各服务职责、执行顺序、数据流及每服务「做哪些操作、变更哪些表」。

### 3.1 总流程与数据流

```
撮合引擎
    │
    ▼
Kafka: exchange-match-result (messageId, symbol, ts, trades, completedOrders)
    │
    ├──────────────────────────────────────────────────────────────────────────┐
    │                                                                          │
    ▼                                                                          ▼
清算服务                                                                   market 服务
(消费 exchange-match-result)                                              (消费同一 topic 或二次 topic)
    │                                                                          │
    │ 产出：清算结果（每笔成交的手续费、买卖双方应收应付、订单完成需退冻结等）        │ 产出：K 线、24h、盘口等
    │                                                                          │
    ▼                                                                          ▼
结算服务                                                                   (写 K 线/行情表或缓存)
(消费 清算结果)
    │
    │ 产出：资金指令（谁+什么币+加/减/解冻多少）
    │
    ▼
资金服务
(消费 资金指令)
    │
    │ 执行：member_wallet 变动、member_transaction 流水、exchange_order 状态与退冻结
    │
    ▼
(可选) 资金完成后发事件 / 消息
    │
    ├──────────────────────────────────────────────────────────────────────────┐
    │                                                                          │
    ▼                                                                          ▼
WebSocket 服务                                                           (行情推送服务)
(消费 订单成交/订单完成 事件 或 直接消费 exchange-match-result)                    (消费成交流，推 K 线/盘口)
    │
    │ 推送：order-trade、order-completed 给对应用户
    │
    ▼
前端 / 客户端
```

说明：

- **清算**：只做「算」，产出结构化清算结果，不直接动钱包与订单表。
- **结算**：根据清算结果生成「资金指令」，不执行划转。
- **资金**：只执行指令，改钱包、流水、订单状态与退冻结。
- **market**：只做行情与盘口聚合，写 K 线等，不碰资金与订单状态。
- **WebSocket**：只做「推」，可订阅资金/订单事件或撮合结果，推给用户。

### 3.2 各服务职责、输入输出、操作与变更表

#### 3.2.1 清算服务

| 项目 | 内容 |
|------|------|
| **职责** | 接收成交流水（trades + completedOrders），计算每笔成交的手续费、买卖双方应收应付、已完全成交订单需退冻结金额；产出**清算结果**（结构化数据），不执行任何资金或订单状态变更。 |
| **输入** | Kafka：exchange-match-result（或专用 topic），消息体含 messageId, symbol, ts, trades, completedOrders。 |
| **输出** | 清算结果（可写表或发 Kafka）：如 清算主表 + 清算明细表（成交 id、买卖方、手续费、应收应付、订单完成退冻金额等）。 |
| **主要操作** | 解析 trades、completedOrders；按规则计算手续费、应收应付、退冻；写清算结果表；可选发 Kafka「清算结果」给结算。 |
| **变更数据表** | **clearing_result**（示例，下同）：清算主表；**clearing_detail**：每笔成交/每笔订单完成的清算明细。不涉及 member_wallet、exchange_order、member_transaction。 |

#### 3.2.2 结算服务

| 项目 | 内容 |
|------|------|
| **职责** | 根据清算结果生成**资金指令**（谁、什么币、加/减/解冻多少），不执行划转。 |
| **输入** | 清算结果（表或 Kafka）。 |
| **输出** | 资金指令列表（可写表或发 Kafka）：如 指令 id、用户、币种、类型（加可用/减冻结/解冻）、金额、关联订单/成交 id 等。 |
| **主要操作** | 读清算结果；生成一条条资金指令；写结算单表/资金指令表；可选发 Kafka「资金指令」给资金服务。 |
| **变更数据表** | **settlement_instruction**（资金指令表）：INSERT；可选 **settlement_bill**（结算单主表）。不涉及 member_wallet、member_transaction。 |

#### 3.2.3 资金服务

| 项目 | 内容 |
|------|------|
| **职责** | 执行资金指令：更新钱包（加可用、减冻结、解冻），记流水，更新订单状态与完成时间；不计算手续费、不产行情。 |
| **输入** | 资金指令（表或 Kafka）；或直接消费 exchange-match-result 时自行从 trades/completedOrders 推导指令并执行（简化形态）。 |
| **输出** | 无对外业务输出；内部写 wallet、transaction、order 等。 |
| **主要操作** | 按指令对 **member_wallet** 做 increaseBalance / decreaseFrozen / thawBalance；对每笔变动写 **member_transaction**；对已完全成交订单更新 **exchange_order**（status, tradedAmount, turnover, completedTime）并执行退冻结。可选写 **exchange_order_detail**（若由资金服务统一写成交明细）。 |
| **变更数据表** | **member_wallet**：UPDATE；**member_transaction**：INSERT；**exchange_order**：UPDATE；可选 **exchange_order_detail**：INSERT。不涉及 K 线、盘口、推送。 |

#### 3.2.4 market 服务（行情/盘口）

| 项目 | 内容 |
|------|------|
| **职责** | 用成交流水更新 K 线、24h 涨跌量、盘口、最新成交等，只做行情聚合与存储，不碰资金与订单状态。 |
| **输入** | Kafka：exchange-match-result 或 专用「成交流水」topic（仅 trades + symbol 等）。 |
| **输出** | 更新后的 K 线、CoinThumb、盘口、最新成交列表等（供接口或推送服务读取）。 |
| **主要操作** | 解析 trades；CoinProcessor.process(trades) 更新内存 K 线与 CoinThumb；通过 MarketHandler 持久化 K 线；pushJob.addTrades 等加入盘口/最新成交队列。 |
| **变更数据表/存储** | **K 线存储**（如 MongoDB 集合或 Redis）：INSERT/UPDATE；**CoinThumb** 等行情缓存；盘口、最新成交缓存。不涉及 member_wallet、exchange_order、member_transaction。 |

#### 3.2.5 WebSocket 服务（推送）

| 项目 | 内容 |
|------|------|
| **职责** | 将「订单部分成交」「订单完全成交」等事件推送给对应用户（WebSocket / Netty），不落库、不计算、不执行资金。 |
| **输入** | 订单成交/订单完成事件（Kafka 或 MQ；或由资金服务/清算服务在关键节点发事件）；或直接消费 exchange-match-result 仅做推送。 |
| **输出** | 向前端/客户端推送：order-trade、order-completed 等。 |
| **主要操作** | 按 memberId、symbol 路由；messagingTemplate.convertAndSend、nettyHandler.handleOrder 等。 |
| **变更数据表** | 无。仅读订单/用户信息用于组包推送。 |

### 3.3 分层下「成交明细、手续费聚合、返佣」归属建议

- **成交明细（exchange_order_detail）**：可在**资金服务**执行资金变动时顺带写入，或由**清算服务**产出「清算明细」后由**资金服务**根据明细写 exchange_order_detail；避免多服务重复写同一张表。
- **手续费聚合（order_detail_aggregation 等）**：属于统计/运营数据，可由**清算服务**写，或由**结算/资金**在写流水时异步写；与「资金正确性」解耦。
- **返佣（reward_record、推荐人流水）**：可在**清算服务**中算好「谁得多少返佣」，作为清算结果一部分，由**结算**生成「推荐人加款」类资金指令，由**资金服务**执行；或仍在资金服务内按当前逻辑在写流水时同步写返佣（简化）。

## 3.4 按 3.1 流程做的收益：更高可用、职责更清晰

若按 **3.1 总流程与数据流** 实现（撮合 → 清算 → 结算 → 资金 串行，market / WebSocket 与清算并行消费），系统会**更高可用、职责更清晰**，原因如下。

### 更高可用

| 点 | 说明 |
|----|------|
| **撮合与下游解耦** | 撮合只发 Kafka（方案 A 下先 WAL 再发），不等待清算、资金、market；下游消费慢只造成 lag，不阻塞撮合。 |
| **故障隔离** | 清算、结算、资金、market、推送 各自消费、各自重试；某一环节故障或变慢，不影响其他环节（如行情可照常更新）。 |
| **独立扩缩容** | 可按 topic 消费 lag 单独扩容某一消费者（如资金服务），无需整体扩容。 |
| **数据不丢** | 清算/结算/资金 均「先落库再发 Kafka 或执行」，未发成功可定时重试，与 3.1 设计一致。 |

### 职责更清晰

| 服务 | 单一职责 | 好处 |
|------|----------|------|
| **market** | 只做行情与盘口（K 线、24h、最新成交等），不碰资金、不写订单状态 | 行情与资金解耦，行情可单独迭代、部署、回滚。 |
| **清算** | 只做「算」：手续费、应收应付、退冻金额，产出清算结果 | 规则集中、易对账、易做清算审计。 |
| **结算** | 只做「指令」：根据清算结果生成资金指令，不执行划转 | 指令可落库、可重放、可对账。 |
| **资金** | 只做「执行」：按指令改钱包、写流水、更新订单状态与退冻结 | 资金变更单一入口，易做风控与审计。 |
| **WebSocket/推送** | 只做「推」：订阅事件或成交流水，推给前端 | 推送与业务逻辑解耦，可独立扩展。 |

### 当前实现与 3.1 的差距（可演进方向）

| 维度 | 3.1 理想 | 当前实现 | 可演进方向 |
|------|----------|----------|------------|
| **消费 match-result** | 清算 + market 分别消费同一 topic | **已实现**：ClearingMatchResultConsumer（group=market-clearing-group）独立消费做清算；ExchangeTradeConsumer 消费做订单+行情+推送。 | — |
| **market 职责** | 只做行情，不写订单、不写成交明细 | 可配置为「仅订单状态+成交明细+行情+推送」或「直接改钱包」；**清算已不在 market 内触发** | 可选：订单状态与成交明细改由资金服务写（见 3.5，当前建议保留在 market）。 |
| **订单状态与成交明细** | 3.1 图中写为资金服务可更新 exchange_order | 当前由 market 的 processMatchResultOrderOnly 写订单状态与 exchange_order_detail | **建议保留在 market**（领域归属、时延、体验更优）；见 3.5。 |
| **资金流水 member_transaction** | 资金服务执行时写 | 当前资金服务只改 member_wallet，未写 member_transaction | 在资金服务执行 INCOME/OUTCOME/FEE/REFUND 时同步写 member_transaction，与 3.2.3 一致。 |
| **推送** | 独立 WebSocket 服务，订阅事件或 match-result | 推送仍在 market 的 handleMatchResult 内（与行情同进程） | 可选：资金服务完成后发「订单成交/完成」事件，独立推送服务消费并推 WebSocket/Netty。 |

结论：**按 3.1 的流程与职责划分，系统确实更高可用、职责更清晰**；当前已具备「清算独立消费 match-result → 结算 → 资金」流水线与 market 仅落订单状态+行情+推送，在此基础上可按上表逐步向 3.1 靠拢（资金写流水、推送独立等）。**订单状态/明细** 是否放到资金服务见下节 3.5。

### 3.5 订单状态与成交明细：放 market 还是放资金服务？

3.1 图中写的是「资金服务执行……exchange_order 状态与退冻结」，是一种**执行一致性**的划分；但**不一定更合理**。两种做法对比如下。

#### 放在资金服务（3.1 写法）的理由

- **执行一口径**：钱包变动、流水、订单完成、退冻结 都在资金服务完成，相当于「结算结果」的执行只在一个服务里完成，对账、审计时只盯资金服务即可。
- **状态一致**：避免「订单已 COMPLETED 但钱还没动」或「钱已动但订单还 TRADING」的中间态（若资金服务在同一个事务里改订单+钱包，则要么全成功要么全失败）。

#### 放在 market（当前实现）往往更合理的理由

| 维度 | 说明 |
|------|------|
| **领域归属** | 订单、成交明细属于**交易/订单域**（exchange_order、exchange_order_detail），应由「交易/行情」侧维护；钱包、流水属于**资金域**，由资金服务维护。按领域划分，**订单状态与成交明细归 market（交易侧）更自然**。 |
| **时延与体验** | market 与撮合同链路、先消费 match-result，可**先落订单状态与成交明细**，用户很快看到「订单已成交」；资金服务在清算→结算→资金之后执行，若订单状态也等资金服务再改，**完成态会明显滞后**，体验更差。 |
| **故障隔离** | 资金服务异常或积压时，若订单状态在 market，至少**订单已完成**可先展示，余额稍后更新；若订单状态在资金服务，会出现「撮合已成交但订单一直显示交易中」，更难以解释。 |
| **读写一致** | 行情、盘口、最新成交 都在 market，订单状态与成交明细也在 market，**查订单/查成交**不必跨服务；若订单状态在资金服务，查询「我的订单是否完成」要依赖资金服务已执行完，耦合更大。 |

#### 小结建议

- **订单状态（COMPLETED、tradedAmount、turnover、completedTime）与成交明细（exchange_order_detail）** 建议保留在 **market** 消费 match-result 时写入，不强行迁到资金服务。
- **资金服务** 只负责：按资金指令改 **member_wallet**、写 **member_transaction**、执行 **退冻结**（REFUND）；不必须承担「更新 exchange_order 状态」。
- 3.1 中「资金服务更新 exchange_order」可视为**可选形态**（强调执行一口径时采用）；当前「market 写订单状态与明细，资金服务只写钱包与流水」**职责清晰、时延更优、领域更顺**，无需为贴 3.1 而迁移订单状态/明细到资金服务。

## 四、当前实现 vs 分层实现 对照小结

| 维度 | 当前实现 | 分层实现（market + 清算 + 结算 + 资金 + WebSocket） |
|------|----------|---------------------------------------------------|
| **消费 exchange-match-result** | **清算**（ClearingMatchResultConsumer，group=market-clearing-group）与 **market**（ExchangeTradeConsumer）各自独立消费同一 topic；资金消费「资金指令」。 | 清算 + market 可同时消费；资金消费「资金指令」。 |
| **成交明细** | market → ExchangeOrderService → exchange_order_detail | 资金服务写（或清算产出、资金写）。 |
| **手续费 / 流水 / 钱包** | market → processOrder → member_wallet、member_transaction | 清算算 → 结算生成指令 → 资金执行 → member_wallet、member_transaction。 |
| **订单状态与退冻结** | market → tradeCompleted / orderRefund | 资金服务按指令更新 exchange_order 并退冻结。 |
| **K 线 / 行情** | market → CoinProcessor、MarketHandler、pushJob | market 服务独立消费成交，只写 K 线/盘口。 |
| **推送** | market → messagingTemplate、nettyHandler | WebSocket 服务独立，订阅事件或成交流水后推送。 |
| **幂等** | processMatchResultIdempotent + processed_match_result_message | 各服务按自身语义做幂等（如资金按指令 id，清算按 messageId）。 |

## 五、撮合与下游解耦：清算/结算消费速度不影响撮合

- **撮合** 只负责产出 MatchResult，通过 **Kafka**（方案 A 下先写 WAL、再异步发 Kafka）下发，**不等待**任何下游（清算、结算、资金、market）消费完成。
- **清算、结算、资金、market、WebSocket** 各自从 Kafka 拉取并处理；处理慢只会造成 topic 积压或消费 lag，**不会反向阻塞撮合**。
- 可能拖慢撮合的只有**本机**：若方案 A 的队列满，撮合线程会在 `put(MatchResult)` 上阻塞（Writer/Sender 跟不上），与下游服务消费速度无关。
