# 用户下单：结算过程

本文档说明**结算服务**的实现：结算输入与输出、处理流程、落库与 Kafka 的关系，以及如何保证**结算结果（资金指令）不丢、可恢复**；并记录改造过程，并给出从订单到交易流水、清算结果、结算消费生成资金指令的完整数据示例。

---

## 一、结算在整体链路中的位置

```
订单 → 撮合 → exchange-match-result
  → Market: 订单/行情/推送落库；Clearing 服务: 清算落库 + exchange-clearing-result
  → 结算（独立 settlement 应用）: SettlementConsumer 消费 exchange-clearing-result
  → 生成资金指令 → 落库 settlement_result → 发 Kafka exchange-fund-instruction
  → 资金/钱包服务（market）消费 exchange-fund-instruction 执行加减款、解冻等
```

结算在**清算之后**，由**独立 settlement 应用**消费清算结果生成**资金指令**，供下游资金服务（market 内 FundInstructionConsumer）执行，不直接写钱包。

---

## 二、结算输入

- **来源**：Kafka topic **exchange-clearing-result**（与清算输出一致；清算先落库再发该 topic）。
- **内容**：单条消息 key=messageId（清算 messageId），value=清算 payload（ClearingResultDTO JSON，含 tradeClearingList、orderRefundList，项内含 memberId、orderId、币种、金额等）。
- **触发**：**settlement 应用**内 **SettlementConsumer** 订阅 exchange-clearing-result（group=market-settlement-group），收到后调用 **SettlementService.processAndPublish(clearingMessageId, clearingPayload)**。
- **幂等**：若表 **settlement_result** 已存在该 messageId，直接 return，不重复生成、不重复落库、不重复发 Kafka。

---

## 三、结算输出

### 3.1 先落库再发 Kafka

| 输出 | 说明 |
|------|------|
| **落库** | 表 **settlement_result**（exchange-core 实体），字段：id, message_id(唯一), symbol, ts, status(PENDING/PUBLISHED/FAILED), payload(JSON 文本), created_at, published_at。 |
| **Kafka** | Topic **exchange-fund-instruction**，单条消息 key=messageId，value=payload（FundInstructionDTO JSON）。 |

顺序：**先写入 settlement_result（status=PENDING），再发 Kafka；发送成功则更新 status=PUBLISHED、published_at；发送失败则更新 status=FAILED**，由定时任务重试发送。

### 3.2 payload 结构（FundInstructionDTO）

- **messageId / symbol / ts**：与清算一致，便于溯源与对账。
- **instructions**：资金指令明细列表。每条 **FundInstructionItem** 含：
  - **memberId**：用户 ID（或平台账户 ID，见下）
  - **orderId**：订单 ID
  - **symbol**：币种（如 BTC、USDT）
  - **amount**：金额（正数=加款/解冻，负数=扣款/扣冻）
  - **type**：INCOME（用户收入）、OUTCOME（用户支出）、FEE（用户支付手续费）、REFUND（退冻结）、**FEE_REVENUE（平台手续费收入）**

**金额语义**：同一币种下 **INCOME 为毛收入（未扣手续费）**，**FEE 为手续费扣款**。资金服务按条执行后，用户该币种净增 = INCOME + FEE（例如 INCOME +0.01、FEE -0.00001 → 净收入 0.00999），避免重复扣费。

由 **tradeClearingList** 拆为：收入（INCOME，毛收入）、支出（OUTCOME）、用户手续费（FEE）、**平台手续费收入（FEE_REVENUE）**；由 **orderRefundList** 拆为：退冻（REFUND）。用户 FEE 与平台 FEE_REVENUE 成对，保证借贷平衡。

### 3.3 清算指令 → 资金指令 对应关系

结算时，每条**清算指令**（清算 payload 中的一项）会生成一条或多条**资金指令**，对应关系如下。

#### （1）来自 tradeClearingList（成交清算项）

每条 **TradeClearingItem**（一笔成交下的一侧：一个 orderId + 一个 memberId + 应收/应付/手续费）按字段生成资金指令，**同一清算项**生成的指令 **memberId、orderId 一致**：

| 清算字段 | 条件 | 生成的资金指令 |
|----------|------|----------------|
| incomeSymbol + incomeAmount + fee | incomeAmount > 0 | **1 条 INCOME**：memberId/orderId 同该项，symbol=incomeSymbol，**amount=incomeAmount+fee（毛收入，正）**，执行后与 FEE 合计得净收入 |
| outcomeSymbol + outcomeAmount | outcomeAmount > 0 | **1 条 OUTCOME**：memberId/orderId 同该项，symbol=outcomeSymbol，amount=-outcomeAmount（负） |
| fee + 收入币种 | fee > 0 | **1 条 FEE**：memberId/orderId 同该项，symbol=该项 incomeSymbol（手续费收取币种），amount=-fee（负） |
| fee + 收入币种 | fee > 0 | **1 条 FEE_REVENUE**：memberId=平台账户，orderId 同该项，symbol 同上，amount=fee（正） |

- 即：**1 条 tradeClearingList 项** → 最多 **4 条**资金指令（INCOME、OUTCOME、FEE、FEE_REVENUE）；若该项 fee=0，则只生成 INCOME、OUTCOME 共 2 条。

#### （2）来自 orderRefundList（退冻结项）

每条 **OrderRefundItem**（一个已完成订单的退冻）生成**1 条**资金指令：

| 清算字段 | 条件 | 生成的资金指令 |
|----------|------|----------------|
| memberId + orderId + coinSymbol + refundAmount | refundAmount > 0 | **1 条 REFUND**：memberId/orderId/coinSymbol 同该项，amount=refundAmount（正） |

- 即：**1 条 orderRefundList 项** → **1 条 REFUND**。

#### （3）对应关系小结

| 清算来源 | 每条清算项 | 生成资金指令条数 | 指令类型 |
|----------|------------|------------------|----------|
| tradeClearingList | 1 项（一侧成交） | 2～4 条 | INCOME（必有则 1）、OUTCOME（必有则 1）、FEE（fee>0 时 1）、FEE_REVENUE（fee>0 时 1） |
| orderRefundList | 1 项（一单退冻） | 1 条 | REFUND |

对账或排查时，可根据资金指令的 **orderId** 反查来自哪一条清算项；同一 orderId 下的多条指令（INCOME/OUTCOME/FEE/FEE_REVENUE）均来自**同一条 tradeClearingList 项**。

**示例（见第八章）**：清算里 tradeClearingList 有 2 项（BO001 买方、SO001 卖方）。  
- 清算项 BO001（买方一侧）→ 生成 4 条资金指令：INCOME(BTC +0.01 毛收入)、OUTCOME(USDT -500)、FEE(BTC -0.00001)、FEE_REVENUE(BTC +0.00001)；用户 BTC 净增 = 0.01 - 0.00001 = 0.00999。  
- 清算项 SO001（卖方一侧）→ 生成 4 条资金指令：INCOME(USDT +500 毛收入)、OUTCOME(BTC -0.01)、FEE(USDT -0.5)、FEE_REVENUE(USDT +0.5)；用户 USDT 净增 = 500 - 0.5 = 499.5。  
- orderRefundList 为空 → 无 REFUND 指令。

## 四、处理流程（消费清算 → 生成资金指令 → 落库 → 发 Kafka）

1. **消费**：SettlementConsumer 消费 **exchange-clearing-result**，取 record.key() 为 messageId，record.value() 为清算 payload。
2. **幂等**：若 **settlementResultRepository.existsByMessageId(messageId)** 为 true，直接 return。
3. **转换**：将清算 payload 解析为 ClearingResultDTO，再经 **clearingToFundInstructions** 转为 FundInstructionDTO（每条清算项拆为 INCOME/OUTCOME/FEE，每条退冻拆为 REFUND）。
4. **落库**：构造 **SettlementResult**（messageId, symbol, ts, status=PENDING, payload=FundInstructionDTO JSON, createdAt），**settlementResultRepository.save**。
5. **发 Kafka**：**kafkaTemplate.send(fundInstructionTopic, messageId, payload)**；成功则更新 status=PUBLISHED、publishedAt；失败则更新 status=FAILED，打日志。
6. **重试**：**RetrySettlementPublishJob** 定时（默认 30s）扫描 status 为 PENDING 或 FAILED 的记录，用其 payload 重新发 Kafka，成功则更新为 PUBLISHED 和 publishedAt。

## 五、如何保证结算结果不丢、可恢复

| 机制 | 说明 |
|------|------|
| **先落库再发 Kafka** | 每条结算结果先写入 **settlement_result** 再发送；发送失败时数据库中已有完整 payload，不会因 Kafka 故障丢失。 |
| **幂等** | 按 messageId（与清算一致）去重：已存在则不再生成、不重复落库、不重复发送。 |
| **失败状态与重试** | 发送失败时 status 置为 FAILED，**RetrySettlementPublishJob** 定时对 PENDING/FAILED 用已有 payload 重发，不重新计算。 |
| **可恢复** | 重启或 Kafka 恢复后，定时任务继续重试；也可从 **settlement_result** 按 messageId 或时间范围查询并人工/脚本重放。 |
| **可对账** | 结算结果持久化在 **settlement_result**，可与清算、资金流水对账。 |

## 六、改造过程记录

### 6.1 exchange-core

- **ClearingResultDTO**：TradeClearingItem、OrderRefundItem 增加 **memberId**，供结算生成资金指令时无需再查订单。
- **ClearingComputeService**：在 computeTradeItem、computeRefundItem 中设置 **memberId**。
- **FundInstructionDTO**：资金指令 DTO，含 messageId, symbol, ts, instructions（List&lt;FundInstructionItem&gt;）；FundInstructionItem 含 memberId, orderId, symbol, amount, type(INCOME/OUTCOME/FEE/REFUND)。
- **SettlementResult** 实体：表 settlement_result，字段 id, message_id(唯一), symbol, ts, status, payload, created_at, published_at。
- **SettlementResultRepository**：existsByMessageId；findByStatusInOrderByIdAsc(List&lt;Status&gt;) 供重试任务使用。

### 6.2 market

- **SettlementService**：clearingToFundInstructions(ClearingResultDTO)→FundInstructionDTO（用户 INCOME/OUTCOME/FEE、平台 FEE_REVENUE、REFUND）；processAndPublish(clearingMessageId, clearingPayload) 幂等 → 解析清算 → 生成资金指令 → 落库 PENDING → 发 Kafka；retryPublishPending 扫描 PENDING/FAILED 重试发送。每笔用户 FEE 同时生成一条 FEE_REVENUE（platformMemberId，正金额）。
- **SettlementConsumer**：@KafkaListener(topics = "exchange-clearing-result")，调用 **settlementService.processAndPublish(record.key(), record.value())**。
- **RetrySettlementPublishJob**：定时（settlement.retry.interval，默认 30000ms）调用 **settlementService.retryPublishPending()**。

### 6.3 配置项（可选）

- **settlement.fund.topic**：资金指令 Kafka topic，默认 **exchange-fund-instruction**。
- **settlement.retry.interval**：重试任务间隔（毫秒），默认 **30000**。
- **settlement.platform.memberId**：平台账户 memberId，用于 FEE_REVENUE 指令；默认 **0**（系统/平台）。

## 七、涉及模块与类汇总

| 模块 | 类/表 | 说明 |
|------|--------|------|
| exchange-core | FundInstructionDTO / FundInstructionItem | 资金指令 DTO 与明细结构。 |
| exchange-core | SettlementResult | 结算结果实体，表 settlement_result。 |
| exchange-core | SettlementResultRepository | JPA 仓库，幂等与重试查询。 |
| exchange-core | ClearingResultDTO（含 memberId） | 清算项带 memberId，供结算使用。 |
| settlement（独立应用） | SettlementService | 清算→资金指令转换、落库、发 Kafka、重试。 |
| settlement | SettlementConsumer | 消费 exchange-clearing-result（group=market-settlement-group）。 |
| settlement | RetrySettlementPublishJob | 定时重试未成功发送的结算结果。 |
| Kafka | exchange-fund-instruction | 资金指令下游 topic。 |

## 八、数据示例：一条订单 → 交易流水 → 清算结果 → 结算资金指令

以下为**单笔成交、买卖双方各一单且该笔成交后两单均完全成交**的简化示例。交易对 **BTC/USDT**，手续费率 0.1%（0.001）。前面阶段与《用户下单-清算过程》一致，此处从清算结果起补充**结算消费后产出的资金指令**。

### 8.1 场景与假设（同清算文档）

- 买方订单 BO001：限价买入 0.01 BTC，价格 50000 USDT（冻结 500 USDT），member_id=100。
- 卖方订单 SO001：限价卖出 0.01 BTC，价格 50000 USDT（冻结 0.01 BTC），member_id=200。
- 撮合产生 1 笔成交：价格 50000，数量 0.01，买卖双方该笔后均完全成交。

### 8.2 订单（exchange_order 表行记录示例）

| 字段 | 买方订单 BO001 | 卖方订单 SO001 |
|------|----------------|----------------|
| order_id | BO001 | SO001 |
| member_id | 100 | 200 |
| symbol | BTC/USDT | BTC/USDT |
| direction | BUY | SELL |
| amount | 0.01 | 0.01 |
| price | 50000 | 50000 |
| traded_amount | 0.01 | 0.01 |
| turnover | 500 | 500 |
| status | COMPLETED | COMPLETED |

### 8.3 成交与资金流水（简要）

- **exchange_order_detail**：2 条（买卖各一），fee 买方 0.00001 BTC、卖方 0.5 USDT。
- **member_transaction**：4 条（买方：+0.00999 BTC、-500 USDT；卖方：+499.5 USDT、-0.01 BTC）。

### 8.4 清算结果（clearing_result 表 + Kafka exchange-clearing-result）

**clearing_result 表行记录示例：**

| 字段 | 示例值 |
|------|--------|
| id | 1 |
| message_id | msg-a1b2c3d4-e5f6-7890-abcd-ef1234567890 |
| symbol | BTC/USDT |
| ts | 1699900000000 |
| status | PUBLISHED |
| payload | （见下方清算 payload JSON） |

```json
{
  "messageId": "msg-a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "symbol": "BTC/USDT",
  "ts": 1699900000000,
  "tradeClearingList": [
    {
      "memberId": 100,
      "orderId": "BO001",
      "direction": "BUY",
      "fee": 0.00001,
      "incomeSymbol": "BTC",
      "incomeAmount": 0.00999,
      "outcomeSymbol": "USDT",
      "outcomeAmount": 500
    },
    {
      "memberId": 200,
      "orderId": "SO001",
      "direction": "SELL",
      "fee": 0.5,
      "incomeSymbol": "USDT",
      "incomeAmount": 499.5,
      "outcomeSymbol": "BTC",
      "outcomeAmount": 0.01
    }
  ],
  "orderRefundList": []
}
```

### 8.5 结算结果（settlement_result 表 + Kafka exchange-fund-instruction）

**settlement_result 表行记录示例：**

| 字段 | 示例值 |
|------|--------|
| id | 1 |
| message_id | msg-a1b2c3d4-e5f6-7890-abcd-ef1234567890 |
| symbol | BTC/USDT |
| ts | 1699900000000 |
| status | PUBLISHED |
| payload | （见下方资金指令 JSON） |
| created_at | 2024-11-14 12:00:02 |
| published_at | 2024-11-14 12:00:03 |

**资金指令 payload（Kafka exchange-fund-instruction 的 value）：**

由结算服务消费上述清算 payload 后生成，每条清算项拆为 INCOME、OUTCOME、FEE（若有），退冻为 REFUND。

```json
{
  "messageId": "msg-a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "symbol": "BTC/USDT",
  "ts": 1699900000000,
  "instructions": [
    // 买方 BTC：INCOME +0.01，FEE -0.00001 → 净增 0.00999 ✓
    {
      "memberId": 100,
      "orderId": "BO001",
      "symbol": "BTC",
      "amount": 0.01,   // INCOME = 毛收入，FEE = 手续费扣款，净增 = INCOME + FEE
      "type": "INCOME"
    },
    {
      "memberId": 100,
      "orderId": "BO001",
      "symbol": "USDT",
      "amount": -500,
      "type": "OUTCOME"
    },
    {
      "memberId": 100,
      "orderId": "BO001",
      "symbol": "BTC",
      "amount": -0.00001,
      "type": "FEE"
    },
    // 卖方 USDT：INCOME +500，FEE -0.5 → 净增 499.5 ✓
    {
      "memberId": 200,
      "orderId": "SO001",
      "symbol": "USDT",
      "amount": 500,
      "type": "INCOME"
    },
    {
      "memberId": 200,
      "orderId": "SO001",
      "symbol": "BTC",
      "amount": -0.01,
      "type": "OUTCOME"
    },
    {
      "memberId": 200,
      "orderId": "SO001",
      "symbol": "USDT",
      "amount": -0.5,
      "type": "FEE"
    },
    {
      "memberId": 0,
      "orderId": "BO001",
      "symbol": "BTC",
      "amount": 0.00001,
      "type": "FEE_REVENUE"
    },
    {
      "memberId": 0,
      "orderId": "SO001",
      "symbol": "USDT",
      "amount": 0.5,
      "type": "FEE_REVENUE"  //FEE_REVENUE: 手续费收入【 REVENUE（营收/总收入）】
    }
  ]
}
```

- **INCOME**：毛收入。买方 BTC 毛收入 0.01，卖方 USDT 毛收入 500；与下条 FEE 执行后，买方 BTC 净收入 = 0.01 - 0.00001 = 0.00999，卖方 USDT 净收入 = 500 - 0.5 = 499.5。
- **OUTCOME**：买方支出 500 USDT（amount 为负）；卖方支出 0.01 BTC（amount 为负）。
- **FEE**：买方手续费 0.00001 BTC（amount 为负）；卖方手续费 0.5 USDT（amount 为负）。
- **FEE_REVENUE**：平台手续费收入，与用户 FEE 成对——买方手续费 0.00001 BTC、卖方手续费 0.5 USDT 归平台（memberId=0，amount 为正）。
- 本示例无退冻，**orderRefundList** 为空，故无 **REFUND** 指令；若有部分成交后完成订单，会出现 type=REFUND、amount 为正的退冻指令。

资金服务消费 **exchange-fund-instruction** 后，可按 instructions 逐条更新 member_wallet（增加可用、扣减冻结等）；FEE_REVENUE 对应平台账户加款或平台收益流水，并与 member_transaction 流水对齐。
