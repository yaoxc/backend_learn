### 问题现象

- 调用 `exchange-api` 的机器人下单接口：
  - `POST /exchange/order/mockaddydhdnskd`
  - 返回：
    - `{"data": "E177364665630086", "code": 0, "message": "success", ...}`
- 观察到这笔订单对应的消息**直接出现在 Kafka 主题 `exchange-order-cancel-success` 中**，看起来像是“刚下单就被系统撤掉了”。

### 结论（为什么会这样）

- **`exchange-api` 本身只会向 `exchange-order-ingress` 发消息，不会直接往 `exchange-order-cancel-success` 发任何东西。**
- 实际链路是：
  1. `OrderController.addOrderMock` 校验参数 → 构造 `ExchangeOrder`；
  2. 调用 `OrderOutboxService.addOrderAndSaveOutbox(uid, order, orderIngressTopic)`：
     - 订单写入 DB（`exchange_order` 表）；
     - 本地消息表 `order_kafka_outbox` 写入一条 `PENDING` 记录；
  3. Controller 拿到 outboxId 后立即调用 `orderOutboxService.trySendById(outboxId)`：
     - 通过 `KafkaTemplate` 把订单 JSON **发送到 Kafka 主题 `exchange-order-ingress`**；
     - 更新 outbox 状态为 `SENT`。
- 之后，**撮合/清算链路**（通常是 `exchange-trade` / `clearing` 等服务）消费 `exchange-order-ingress`，在自己的规则里：
  - 如果认为订单无效（活动时间不符合、价格超范围、余额不足、风控触发、内部异常兜底等），会直接将该订单视为“需要撤销”，并往 `exchange-order-cancel` / `exchange-order-cancel-success` 之类的主题发出“撤单成功事件”。
- 因此，**“订单数据直接进入 `exchange-order-cancel-success`”说明的是：撮合/清算侧在消费 `exchange-order-ingress` 后，立即把这笔订单判定为要撤单，并对外广播撤单成功事件，而不是 `exchange-api` 把 topic 配错。**

### 常见触发原因（需要在撮合/清算侧排查）

- **交易对 / 活动规则限制：**
  - `exchange_coin.publishType` 为抢购 / 分摊模式（`QIANGGOU` / `FENTAN`），当前时间处于活动前、活动中或清盘期间，对 BUY/SELL 或市价单有严格限制；
  - 下单价格不在 `minSellPrice` / `maxBuyPrice` / `publishPrice` 等允许范围内。
- **账户 / 风控：**
  - 对应 `uid` 在撮合系统中可用余额不足；
  - 订单触发撮合侧风控规则，被直接打为无效。
- **撮合内部异常兜底：**
  - 撮合在处理该订单时抛出异常，采用“强制撤单 + 发 cancel-success”作为异常处理策略。

### 排查建议

- 在 `exchange-api`：
  - 确认 `OrderOutboxService.trySendById` 返回成功，且 outbox 状态为 `SENT`；
  - 在 Kafka 中订阅 `exchange-order-ingress`，确认消息内容与下单参数一致。
- 在撮合/清算服务（`exchange-trade` / `clearing` 等）：
  - 用 `orderId = E177364665630086` 搜索日志；
  - 查看从收到 `exchange-order-ingress` 到发出 `exchange-order-cancel-success` 之间的具体决策和错误日志，找到是哪一条规则或异常导致“立即撤单”。
