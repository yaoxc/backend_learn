# 撮合结果可靠投递方案（业界主流）

目标：**消息可靠发送（不丢、不部分成功），且不阻塞撮合热路径（不影响撮合性能）**。

---

## 一、设计原则（业界常见）

1. **撮合热路径不碰网络**：撮合线程只做「写出到本地」或「写入进程内队列」，不直接调 Kafka。
2. **先落盘再发 Kafka**：用本地 WAL（Write-Ahead Log）或等效持久化做缓冲，进程重启后仍可续发。
3. **单条消息原子**：一次撮合产生的「成交 + 订单完成」打成一个 payload 发一条 Kafka 消息，避免「exchange-trade 成功、exchange-order-completed 失败」的部分成功。
4. **独立发送线程**：由后台线程从 WAL/队列读数据、发 Kafka、做重试与进度推进，与撮合线程解耦。

---

## 二、整体架构

```
                    ┌─────────────────────────────────────────────────────────┐
                    │                    CoinTrader (撮合引擎)                  │
                    │  撮合 → 产出 trades + completedOrders                     │
                    └───────────────────────────┬─────────────────────────────┘
                                                 │ 只写本地，不调 Kafka
                                                 ▼
                    ┌─────────────────────────────────────────────────────────┐
                    │  MatchResultPublisher（接口）                              │
                    │  publish(MatchResult)  ← 一次撮合一条结果                  │
                    └───────────────────────────┬─────────────────────────────┘
                                                 │
         ┌───────────────────────────────────────┼───────────────────────────────────────┐
         │ 实现一：内存队列 + 后台写 WAL + 后台发 Kafka（推荐）                            │
         │                                                                               │
         │   publish() → 放入 ArrayBlockingQueue (有界)  → 立即 return                   │
         │                      ↑                                                        │
         │              后台 Writer 线程：queue.take() → 追加写 WAL 文件                   │
         │              后台 Sender 线程：读 WAL → 发 Kafka → 推进 offset/删除已发        │
         └─────────────────────────────────────────────────────────────────────────────┘
```

- **撮合线程**：调用 `publish(MatchResult)`，仅把结果放入内存队列（或直接追加 WAL），然后立刻返回，**不等待 Kafka**。
- **Writer 线程**（若用队列）：从队列取 MatchResult，顺序追加到本地 WAL 文件，保证宕机后仍有持久化。
- **Sender 线程**：从 WAL 按序读取，向 Kafka 发送（带重试），发送成功后推进 WAL 消费位点或归档已发记录。

---

## 三、单条消息原子：MatchResult 结构

一次撮合结果打成**一条** Kafka 消息，避免「只成功一半」。

```json
{
  "symbol": "BTC/USDT",
  "ts": 1699999999999,
  "trades": [
    { "buyOrderId": "a1", "sellOrderId": "b1", "price": 50000, "amount": 0.1, "buyTurnover": 5000, "sellTurnover": 5000, "direction": "BUY", "time": 1699999999999 }
  ],
  "completedOrders": [
    { "orderId": "a1", "tradedAmount": 0.1, "turnover": 5000, "direction": "BUY", "memberId": 100, ... }
  ]
}
```

- **一个 topic**：如 `exchange-match-result`。
- **下游**：一条消息在一个事务里完成：先按 `trades` 写成交明细、改钱包、写流水；再按 `completedOrders` 更新订单状态并退冻结。要么整条成功，要么整条失败，**不会出现「成交写了、订单未完成」的部分成功**。

若单条消息体积过大（例如单次撮合 > 1000 笔成交），可约定单条消息最多 N 条 trade + 对应 completedOrders，或按 symbol 分片 WAL，逻辑不变。

---

## 四、实现方式选型

### 4.1 方案 A：内存队列 + 本地 WAL + 后台 Sender（推荐）

| 步骤 | 说明 |
|------|------|
| 1. 撮合产出 | 构造 `MatchResult(trades, completedOrders)`，调用 `publish(matchResult)`。 |
| 2. publish() | 将 MatchResult 放入**有界** BlockingQueue（如 capacity 10000），队列满则阻塞或丢弃并告警（按业务选择）。**不写 Kafka**，立即 return。 |
| 3. Writer 线程 | 从 queue.take() 取出，**顺序追加**到本机 WAL 文件（按 symbol 分文件或单文件+symbol 前缀）。可每 N 条或每 T ms 批量 fsync，平衡性能与持久化。 |
| 4. Sender 线程 | 顺序读 WAL，每条记录发一条 Kafka 到 `exchange-match-result`，带重试与退避；发送成功后推进 WAL 消费位点（或标记已发、定期截断）。 |
| 5. 重启恢复 | 进程启动时 Sender 从上次位点继续读 WAL 发 Kafka，直到追上最新。 |

**优点**：热路径只有一次 queue.put()，延迟极低；WAL 保证宕机不丢；单条消息原子，无部分成功。  
**缺点**：需管理 WAL 文件与位点、磁盘空间。

### 4.2 方案 B：仅内存队列 + 后台 Sender（无 WAL）

- publish() 只写内存队列；Sender 线程从队列取出发 Kafka。
- **优点**：实现最简单。  
- **缺点**：进程宕机时队列中未发送的数据丢失，仅适合可接受「宕机丢失未发撮合结果、依赖上游重放」的场景；一般不推荐做唯一方案。

### 4.3 方案 C：撮合侧只写 WAL，不设队列

- 撮合线程直接**同步追加** WAL（顺序写，可批量刷盘），不经过队列。
- Sender 线程独立读 WAL 发 Kafka。
- **优点**：无队列满、背压问题；WAL 即唯一持久化。  
- **缺点**：若每次撮合都 sync，可能带来一定延迟；可通过「每 N 条或每 T ms 批量 sync」控制。

## 五、与当前实现的对比与改造要点

| 项目 | 当前 | 本方案 |
|------|------|--------|
| 热路径 | 同步发 Kafka（含重试），阻塞撮合线程 | 只写队列或只写 WAL，不碰 Kafka |
| 可靠性 | 重试 + 日志；宕机后未发即丢 | WAL 持久化，重启续发 |
| 部分成功 | 两条消息（exchange-trade / exchange-order-completed）可能只成功一条 | 一条消息包含 trades + completedOrders，下游单事务处理 |
| 下游 | 两个 topic 两个消费者 | 一个 topic 一个消费者，按 MatchResult 一次事务落库 |

**改造要点**：

1. **CoinTrader**：撮合结束后不再直接调 `handleExchangeTrade` + `orderCompleted`，改为构造 `MatchResult` 并调用 `MatchResultPublisher.publish(matchResult)`。
2. **新增 MatchResultPublisher 实现**：内存队列 + WAL Writer 线程 + WAL Sender 线程（或选方案 B/C）。
3. **新增/调整 Kafka topic**：如 `exchange-match-result`，payload 为上述 MatchResult JSON。
4. **market 侧**：新增或改造消费者，消费 `exchange-match-result`，在**一个事务**内：先按 `trades` 调 `processExchangeTrade` 逻辑（或等价），再按 `completedOrders` 调 `tradeCompleted`；若需兼容旧逻辑，可从 MatchResult 里拆出 trades 与 completedOrders 分别调现有 service，但务必同库同事务。

## 六、小结（可直接作结论）

- **消息可靠**：撮合结果先写本地 WAL（或至少进持久化队列），再由独立 Sender 发 Kafka，失败重试、重启续发。
- **不影响撮合性能**：撮合线程只写内存队列或顺序追加 WAL，不等待网络；Kafka 发送在后台线程完成。
- **无部分成功**：一次撮合对应一条 Kafka 消息（trades + completedOrders），下游单事务处理，避免「成交到了、订单完成没到」导致重启重复撮合。

按上述方式即可在「业界主流」思路上，同时保证**消息可靠发送**和**撮合性能**。

## 八、业界主流：订单簿恢复与「避免重启重复撮合」

### 8.1 问题

若撮合结果先写 WAL、再异步发 Kafka，而**订单簿在重启时从业务库（DB）按 TRADING 状态加载**，会出现：

- 已在本机撮合、MatchResult 尚在队列或 WAL 未发到 Kafka → 业务库仍为 TRADING、未写 exchange_order_detail；
- 重启后从 DB 加载这些订单进订单簿，会**再次参与撮合**，造成同一订单被撮合两次、重复成交与资金风险。

### 8.2 业界常见做法：事件日志为单一事实来源、订单簿由回放恢复

主流 CEX 撮合引擎普遍采用「**事件日志 + 确定性回放**」，保证重启后订单簿与撮合结果一致，**不从业务库恢复订单簿**：

| 要点 | 说明 |
|------|------|
| **单一事实来源** | 撮合侧维护**订单事件日志**（Order Event Log）：每笔订单的接收、撤单、以及可选地撮合结果（或单独 Match Result Log），**只追加、不覆盖**。订单簿状态由该日志**回放**得到，不依赖业务库的 order 表状态。 |
| **Gateways → Sequencer → Matching Engine** | 订单经网关、** Sequencer 赋予全局单调递增序号**，再进入撮合；同一交易对严格按序号处理，保证**确定性**：同一日志回放得到相同订单簿与相同撮合结果。 |
| **重启恢复** | 从**订单事件日志**（及已持久化的 offset/snapshot）回放，在内存中重建订单簿；**不**从业务库查 TRADING 订单灌入。这样「已撮合但未发 Kafka」的订单在日志里已经体现为「已从订单簿移除/已成交」，不会再次进入订单簿，**从根源上避免重复撮合**。 |
| **业务库角色** | 订单表、成交表、状态更新由**下游消费撮合结果**后异步写库；撮合引擎不读业务库做恢复，业务库允许滞后，与撮合侧日志一致即可。 |

### 8.3 典型实现形态

- **形态一：订单日志 + 撮合结果日志分离**  
  - 订单日志：每笔 order 入参、cancel 等顺序追加；重启时只回放订单日志到订单簿（到当前 offset），得到「当前挂单」与每单剩余量。  
  - 撮合结果日志：即当前方案 A 的 WAL，只负责发 Kafka；不参与订单簿重建。  
  - 关键：**订单簿恢复只用订单日志**，订单日志在「撮合产生结果」时同步或先于写一条「match applied」或依赖「订单日志里订单的剩余量已在撮合时更新」——即**订单状态（含剩余量）的权威在撮合侧日志**，而不是 DB。

- **形态二：统一事件日志（Order + Match 同序）**  
  - 所有事件（订单接入、撤单、撮合产出）进**同一条有序日志**，同一 sequencer 赋序。  
  - 重启时整条日志回放：订单事件更新订单簿，撮合事件只用于「已发送位点」之后继续发下游，**不**再驱动订单簿（订单簿已由订单事件正确更新）。  
  - 同一份日志既恢复订单簿，又恢复「哪些 MatchResult 已发」的位点，无重复撮合。

- **形态三：快照 + 增量日志**  
  - 定期将订单簿快照 + 日志 offset 落盘；重启时加载最近快照，再回放该 offset 之后的订单/撮合事件。  
  - 仍满足「订单簿来自撮合侧自身日志/快照，不读业务库 TRADING」。

### 8.4 与当前实现的差异与可选演进

| 维度 | 当前实现 | 业界主流思路 |
|------|----------|--------------|
| **撮合结果持久化** | 有：队列 + WAL，发 Kafka 后推进 offset | 同左或统一事件日志 |
| **订单簿恢复** | 从 DB 查 TRADING 订单 + exchange_order_detail，调用 `trader.trade(tradingOrders)` 灌入 | 从**撮合侧订单事件日志**（或统一日志）回放恢复，**不从 DB 恢复订单簿** |
| **重复撮合风险** | 存在：已撮合未下发的订单在 DB 仍 TRADING，重启后被再次加载、再次撮合 | 无：已撮合订单在日志中已体现为「已成交/已移除」，回放后不会再次进簿 |

**可选演进方向**（需按业务与运维成本选型）：

1. **短期**：重启前尽量让 Sender 把 WAL 发完、队列排空；或低峰重启；监控队列与 WAL 积压，降低「未下发就重启」的概率。  
2. **中期**：引入**撮合侧订单日志**（每笔 order/cancel 先追加到本地日志再进订单簿），重启时**只**用该日志回放恢复订单簿，不再从 DB 拉 TRADING 订单；DB 仅作业务查询与下游异步更新。  
3. **长期**：订单与撮合结果同序进**统一事件日志**，用 snapshot + 增量回放恢复订单簿与发送位点，与业界「事件溯源 + 确定性回放」一致。
