## 资金指令动账场景设计：扫链、归集、提现等

本文只聚焦 **非撮合类资金场景**（扫链充值、C2C 充值、人工充值、提现、归集、冷热再平衡等），说明：

- **场景**：资金如何变动（可用 / 冻结）；
- **入口统一方案**：如何在各业务侧构造 `FundInstructionDTO`，通过 Kafka 交给资金服务统一动账与落流水；
- 与现有 **撮合 → 清算 → 结算 → 资金服务** 链路的关系。

---

## 一、总体思路：多入口业务 → 唯一资金入口

- **统一动账入口**：所有资金变动（包括撮合成交、扫链、归集、提现、人工等），最终都应通过资金服务的：
  - `FundInstructionExecuteService.processAndExecute(messageId, payload)`
  - 由其完成：
    - 幂等控制（`ProcessedFundInstruction`）；
    - 钱包余额/冻结变更（`member_wallet`）；
    - 资金流水落库（`member_transaction`，带 `refType` / `refId`）。

- **多种资金指令生产方**：
  - 撮合类：撮合 → 清算 → 结算 → 产出 `FundInstructionDTO` → Kafka；
  - 非撮合类：**各业务服务（扫链、归集、提现等）直接产出 `FundInstructionDTO`** → Kafka。

> 关键点：**扫链、归集、提现等不需要绕一圈「清算/结算」**，但要统一用 `FundInstructionDTO + FundInstructionExecuteService` 作为动账入口。

---

## 二、资金指令结构回顾（简版）

### 2.1 顶层字段

- `messageId`：幂等键，推荐用业务唯一 ID（任务 ID / 记录 ID 等）；
- `refType`：业务类型，取 `MemberTransaction.REF_TYPE_*` 中的一个，如：
  - `DEPOSIT_CHAIN` / `DEPOSIT_C2C` / `ADMIN_RECHARGE`；
  - `WITHDRAW` / `SWEEP` / `REBALANCE` 等；
- `refId`：业务主键，如充值记录 ID、提现单号、归集批次 ID 等；
- `instructions`：一组 `FundInstructionItem`，每条描述一个「用户 + 币种 + 方向」的资金变更。

### 2.2 明细字段（FundInstructionItem）

- `memberId`：资金变动所属用户；
- `symbol`：币种；
- `amount`：金额（一般传正数，由 `InstructionType` 决定加减方向）；
- `instructionType`：
  - `INCOME`：可用增加；
  - `OUTCOME`：冻结扣减（成交消耗冻结、归集冻结等）；
  - `FEE`：用户支付手续费（从可用扣）；
  - `REFUND`：退冻结（冻结解冻为可用）；
  - `FEE_REVENUE`：平台手续费收入；
- `orderId` / `itemRefId`：该条指令的业务主键（优先级高于 DTO 级的 `refId`），会写入流水表的 `refId`。

资金服务会根据 **refType + instructionType** 映射到具体的 `TransactionType`，并写入 `member_transaction`。

## 三、扫链充值（链上充值）

### 3.1 资金行为

- 用户从外部地址往平台链上充值；
- 区块确认后，平台为用户 **增加可用余额**（无冻结环节）。

### 3.2 指令方案

- **refType**：`MemberTransaction.REF_TYPE_DEPOSIT_CHAIN`；
- **refId**：充值记录 ID 或区块扫描记录 ID；
- **指令列表**：单条 INCOME 指令。

示例（伪代码）：

```java
FundInstructionDTO dto = new FundInstructionDTO();
dto.setMessageId(depositRecord.getTxId()); // 或内部唯一 ID
dto.setRefType(MemberTransaction.REF_TYPE_DEPOSIT_CHAIN);
dto.setRefId(String.valueOf(depositRecord.getId()));

FundInstructionDTO.FundInstructionItem item = new FundInstructionDTO.FundInstructionItem();
item.setMemberId(depositRecord.getMemberId());
item.setSymbol(depositRecord.getSymbol());
item.setAmount(depositRecord.getAmount());
item.setInstructionType(FundInstructionDTO.InstructionType.INCOME);

dto.getInstructions().add(item);
// 序列化后发到资金 Kafka topic
```

### 3.3 资金服务侧效果

- 钱包：`increaseBalance`，用户可用余额增加；
- 流水：`refType=DEPOSIT_CHAIN`，`type=RECHARGE_CHAIN`，`amount>0`。

## 四、C2C / 法币充值（走资金服务）

> 若你希望 C2C 充值也统一走资金服务而非直接改钱包。

### 4.1 资金行为

- 用户通过 C2C/法币渠道购币，完成后给用户加币（通常无冻结环节）。

### 4.2 指令方案

- **refType**：`MemberTransaction.REF_TYPE_DEPOSIT_C2C`（或 `REF_TYPE_OTC`，视业务口径而定）；
- **refId**：C2C 订单 ID 或划拨记录 ID；
- **指令列表**：单条 INCOME 指令。

资金服务效果类似扫链，只是 `TransactionType` 变为 `RECHARGE_C2C` 或配合 `OTC_BUY` 使用。

---

## 五、提现动账（统一走资金服务）

提现通常拆为几个阶段：**申请冻结、链上出金成功、失败退回**。按你现在的设计，推荐将**实际资金变更**统一搬到资金服务，通过资金指令驱动。

### 5.1 场景 A：提现申请（可用 → 冻结）

- 当前可以保留在旧钱包服务中直接做，也可以改成资金指令方式：
  - **可选方案**：在「提现申请通过」处发一条 `WITHDRAW` + OUTCOME/REFUND 指令，让资金服务统一处理冻结/解冻。
  - 若暂时不重构，可先保持旧实现，这一段可作为过渡。

> 若要完全统一入口，建议后续将“申请冻结”也改为：`refType=WITHDRAW` + `InstructionType.OUTCOME`（或自定义一条专门的冻结指令含义），对应 `WITHDRAW_FREEZE`。

### 5.2 场景 B：提现成功（链上打币成功）

**资金行为**：
- 用户冻结余额减少（已成功出金）；
- 可用余额不变（已在申请时从可用转成冻结）；
- 写一条提现出账流水。

**指令方案**：

- **refType**：`MemberTransaction.REF_TYPE_WITHDRAW`；
- **refId**：提现记录 ID；
- **指令**：
  - 一条 `InstructionType.OUTCOME`，`amount = 提现金额 + 手续费（若冻结中包含手续费）`；
  - 若手续费单独从可用扣，则再加一条 `InstructionType.FEE`。

资金服务映射：

- 钱包：`decreaseFrozen` / `deductBalance`；
- 流水：`type = WITHDRAW`，`refType = WITHDRAW`，`amount < 0`，`fee > 0`（若有）。

### 5.3 场景 C：提现失败 / 取消（解冻）

**资金行为**：
- 提现金额从冻结退回可用；
- 写「提现解冻」流水。

**指令方案**：

- **refType**：`MemberTransaction.REF_TYPE_WITHDRAW`；
- **refId**：提现记录 ID；
- **指令**：一条 `InstructionType.INCOME`，`amount = 解冻金额`。

资金服务映射：

- 钱包：`thawBalance`（通过 REFUND 或 INCOME 视你设计；当前实现中 `WITHDRAW + INCOME` → `WITHDRAW_UNFREEZE`）；\n- 流水：`type = WITHDRAW_UNFREEZE`，`amount > 0`，表示提现解冻入账。

## 六、归集（sweep）动账

### 6.1 资金行为

- 从「热钱包」用户地址（或某个归集账户）**冻结/划出**资产；
- 归集完成后，在目标账户记一条归集到账/解冻流水（视你是否需要在用户维度体现）。

### 6.2 指令方案（以用户钱包维度为例）

- **refType**：`MemberTransaction.REF_TYPE_SWEEP`；
- **refId**：归集批次 ID 或任务 ID；

可以拆成两类指令：

1）**归集冻结/划出（OUTCOME）**  
   - `InstructionType.OUTCOME`，`amount = 归集金额`；  
   - 资金服务映射为：`SWEEP + OUTCOME` → `SWEEP_FREEZE`；
   - 钱包端可以是「可用 → 冻结」或「冻结 → 外部」，取决于你在 `MemberWalletService` 的具体实现，资金服务这边只负责调用相应方法。

2）**归集解冻/到账（INCOME）**  
   - `InstructionType.INCOME`，`amount = 解冻/到账金额`；  
   - 映射为：`SWEEP + INCOME` → `SWEEP_UNFREEZE`；
   - 若目标账户是平台统一归集地址，可用 `memberId = 平台归集账户 ID`。

## 七、冷热再平衡（rebalance）

### 7.1 资金行为

- 在「热钱包账户」与「冷钱包账户」之间做资产调拨；
- 总资产不变，只是不同账户余额改变。

### 7.2 指令方案

- **refType**：`MemberTransaction.REF_TYPE_REBALANCE`；
- **refId**：再平衡任务 ID；
- **指令列表**：
  - 热 → 冷：对热账户发 OUTCOME 指令；
  - 冷 → 热：对冷账户发 OUTCOME 指令；
  - 收到方各自发 INCOME 指令。

资金服务映射：

- 所有指令 `refType=REBALANCE` → 流水 `TransactionType.REBALANCE`；
- 用户维度可能是「平台内部账户之间」的划转，方便运维对账。

---

## 八、总结与落地步骤

1. **统一入口**：所有非撮合资金场景（扫链、归集、提现、人工、C2C 充值等）都不再直接改 `member_wallet`，而是在各自业务模块内构造 `FundInstructionDTO`，写入 Kafka。 \n2. **规范 refType/refId**：\n   - 扫链：`DEPOSIT_CHAIN + 充值记录 ID`；\n   - C2C 充值：`DEPOSIT_C2C + 订单 ID`；\n   - 提现：`WITHDRAW + 提现单 ID`；\n   - 归集：`SWEEP + 归集批次 ID`；\n   - 再平衡：`REBALANCE + 任务 ID`。\n3. **指令颗粒度**：每条 `FundInstructionItem` = 一个用户 + 一个币种 + 一个方向（INCOME/OUTCOME/FEE/REFUND/FEE_REVENUE）。  \n4. **资金服务负责**：幂等、钱包余额/冻结变更、流水落库、与 `TransactionType` 的映射。  \n\n做到以上四点后，你就有了：\n\n- **多入口业务统一通过资金指令落地**；\n- **一套清晰的流水模型（TransactionType + refType/refId）**；\n- **单点可观察的资金执行服务（FundInstructionExecuteService）**，利于审计和排错。\n+
