# 流水记录：资金解冻与资金指令执行

本文说明**资金指令的解析**与**余额、流水入账逻辑**，基于当前 `FundInstructionExecuteService` 与 `FundInstructionDTO` 实现。

---

## 1. 整体资金链路（分层）

从资金账户视角，一笔订单的完整资金轨迹是：

1. **用户下单（买/卖）**
   - 买单：冻结计价币（如 USDT）；卖单：冻结标的币（如 BTC）。
   - 资金行为：**可用 → 冻结**，总资产不变。
   - 由**下单/撮合服务**调用钱包做冻结，并可写一条「下单冻结」流水（`TransactionType.EXCHANGE_FREEZE`）。

2. **撮合成交后**
   - 清算服务根据成交结果产出**资金指令**（`FundInstructionDTO`），发往 Kafka。
   - **资金服务**消费消息，解析指令后对钱包做：**扣减冻结（OUTCOME）**、**增加可用（INCOME/REFUND）**、**扣费/手续费收入（FEE/FEE_REVENUE）**，并写 `member_transaction` 流水。

> 直观理解：**下单阶段只锁钱，撮合+清算阶段才真正“花钱/收钱/退钱”**；资金服务只关心“已发生的清算结果”，不关心撮合细节。

---

## 2. 资金指令结构（FundInstructionDTO）

资金指令由上游（如 settlement）构造并写入 Kafka payload，资金服务反序列化为 `FundInstructionDTO`。

### 2.1 DTO 顶层字段

| 字段 | 含义 | 资金服务侧行为 |
|------|------|----------------|
| messageId | 与清算 messageId 一致，幂等与溯源 | 幂等键：已 PROCESSED 则直接 return |
| symbol | 币种（可选，实际以 item 为准） | - |
| refType | 业务类型 | 为空时默认 `ORDER`；写流水时原样写入 `member_transaction.refType` |
| refId | 业务主键（订单号/存款单号/提现单号等） | 为空时用 messageId；每条指令可被 item 级 refId 覆盖（见下） |
| instructions | 指令明细列表 | 逐条执行，见 3、4 节 |

### 2.2 指令明细（FundInstructionItem）

| 字段 | 含义 |
|------|------|
| memberId | 用户 ID |
| symbol | 币种 |
| amount | 金额（正数；支出类在业务侧传正数，执行时按 InstructionType 做加减） |
| instructionType | 指令类型：INCOME / OUTCOME / FEE / REFUND / FEE_REVENUE |
| orderId | 订单号（refType=ORDER 时优先作为流水 refId） |
| itemRefId | 本条业务主键（覆盖 DTO 级 refId） |

---

## 3. 资金指令解析与执行流程

入口：`FundInstructionExecuteService.processAndExecute(messageId, payload)`。

### 3.1 解析与幂等

1. **幂等判断**  
   若已存在 `ProcessedFundInstruction.messageId` 且 `status=PROCESSED`，直接 return，不再执行任何钱包与流水操作。

2. **落库 PENDING**  
   若该 messageId 无记录，则先插入一条 `ProcessedFundInstruction`，`status=PENDING`，`payload=payload`；若已有记录（如上次失败），则以**库中 payload** 为准执行，保证重试数据一致。

3. **解析 DTO**  
   `FundInstructionDTO dto = JSON.parseObject(payloadToUse, FundInstructionDTO.class)`。  
   若 `dto == null` 或 `dto.getInstructions() == null`，将记录标为 FAILED 并 return。

4. **refType / refId**  
   - `refType = dto.getRefType()`，为空则 `MemberTransaction.REF_TYPE_ORDER`。  
   - `refId = dto.getRefId()`，为空则 `messageId`。  
   二者在整条消息内共用，每条指令的流水 refId 再按「3.3 条级 refId」规则解析。

5. **逐条执行**  
   对 `dto.getInstructions()` 中每一条调用 `executeOne(item, refType, refId, messageId)`。  
   全部成功后：`record.status=PROCESSED`，`record.processedAt=now`；任一步异常则：`record.status=FAILED`，事务回滚，由重试任务稍后再次调用 `processAndExecute`。

### 3.2 单条指令：余额操作（executeOne）

先校验：`amount == null` 或 `amount == 0` 则 return；查不到 `MemberWallet` 抛异常，整批回滚。

**本条流水 refId**：`itemRefId = resolveRefId(item, refType, refId, messageId)`，优先级：  
`item.itemRefId` >（refType=ORDER 时 `item.orderId`）> `refId` > `messageId`。

然后按 `item.getInstructionType()` 分支：

| InstructionType | 钱包操作 | 说明 |
|-----------------|----------|------|
| **INCOME** | `increaseBalance(wallet.getId(), amount)` | 增加可用余额 |
| **OUTCOME** | `decreaseFrozen(wallet.getId(), amount.abs())` | 扣减冻结（成交占用冻结部分） |
| **FEE** | `deductBalance(wallet, amount.abs())` | 从可用扣手续费，失败抛异常 |
| **FEE_REVENUE** | `increaseBalance(wallet.getId(), amount)` | 平台手续费收入，增加可用 |
| **REFUND** | `thawBalance(wallet, amount)` | 未成交部分冻结解冻为可用 |
| **default** | 无 | 仅打 warn 日志 |

执行完钱包操作后，调用 `saveTransactionIfMapped(..., refType, itemRefId)`：若根据 refType + instructionType 映射到某个 `TransactionType`，则写一条流水（见下节）。

## 4. 流水入账逻辑（refType + InstructionType → TransactionType）

流水是否写入、以及 `member_transaction.type` 取何值，由 **refType + InstructionType** 共同决定，映射在 `getTransactionType(refType, instructionType)` 中。

### 4.1 映射规则汇总

- **返回 null**：不写流水（如 ORDER 的 OUTCOME，只扣冻结不单独记一条流水）。
- **返回某 TransactionType**：写一条流水，`amount/fee` 由执行侧传入，`refType`、`refId` 使用本条解析得到的 refType 与 itemRefId。

| refType | InstructionType | TransactionType（流水类型） |
|---------|-----------------|-----------------------------|
| ORDER | INCOME / FEE / FEE_REVENUE / REFUND | EXCHANGE |
| ORDER | OUTCOME | **null**（不记流水） |
| DEPOSIT | INCOME | RECHARGE |
| DEPOSIT_CHAIN | INCOME | RECHARGE_CHAIN |
| DEPOSIT_C2C | INCOME | RECHARGE_C2C |
| ADMIN_RECHARGE | INCOME | ADMIN_RECHARGE |
| WITHDRAW | OUTCOME / FEE | WITHDRAW |
| WITHDRAW | INCOME | WITHDRAW_UNFREEZE（提现取消解冻） |
| SWEEP | INCOME | SWEEP_UNFREEZE |
| SWEEP | OUTCOME | SWEEP_FREEZE |
| REBALANCE | 任意 | REBALANCE |
| 其他未枚举 refType | - | 按 ORDER 处理（OUTCOME 不记，其余 EXCHANGE） |

### 4.2 各指令类型下的流水 amount / fee

执行侧传入 `saveTransactionIfMapped` 的金额与手续费（与 3.2 对应）：

| InstructionType | 传入 amount | 传入 fee | 说明 |
|-----------------|-------------|----------|------|
| INCOME | amount（正） | 0 | 收入 |
| OUTCOME | -\|amount\|（负） | 0 | 冻结减少，记负金额 |
| FEE | -\|amount\|（负） | \|amount\| | 用户手续费支出 |
| FEE_REVENUE | amount（正） | 0 | 平台手续费收入 |
| REFUND | amount（正） | 0 | 退冻 |

流水表 `member_transaction` 会写入：`memberId, symbol, amount, type, fee, refType, refId`（及 realFee/discountFee 等），便于按业务与订单维度查询、对账。

---

## 5. 下单冻结与清算解冻的责任分工

- **下单/撮合服务**：负责“可用 → 冻结”，并可写「下单冻结」流水（`EXCHANGE_FREEZE`）。
- **清算服务**：只产出清算结果与资金指令 DTO，不操作钱包、不写流水。
- **资金服务**：根据资金指令做“扣冻结、加可用、扣费、退冻”，并写「成交相关」流水（`EXCHANGE` / 或按 refType 的 RECHARGE、WITHDRAW、SWEEP 等）。

这样分层后：  
- 下单逻辑只负责锁钱与冻结流水；  
- 资金服务只关心“已发生的清算结果”，统一解冻、划转、记流水；  
- 用户流水可通过 `symbol + type + refType + refId` 查看并和订单/提现/充值等业务对齐。

---

## 6. 上游对 refType/refId 的填充

资金服务**已完整支持** refType/refId：从 DTO 读取并写入每条流水。上游只需在构造 `FundInstructionDTO` 时按业务设置：

- **dto.refType**：如 `ORDER`、`WITHDRAW`、`DEPOSIT`、`DEPOSIT_CHAIN`、`SWEEP`、`REBALANCE` 等（见 `MemberTransaction.REF_TYPE_*`）。
- **dto.refId**：业务主键（订单号、提现单号、存款单号、归集批次 ID 等）；可为空，资金服务会回填 messageId 或 item.orderId（ORDER 场景）。
- **item.orderId**（refType=ORDER 时）：会作为该条流水的 refId，便于与 exchange_order 关联。
- **item.itemRefId**（可选）：覆盖本条流水的 refId，用于单条指令对应不同业务单的场景。

当前结算（SettlementService）已设置 `refType=ORDER`、`refId=messageId`；订单场景下每条 item 的 orderId 会作为流水 refId。
