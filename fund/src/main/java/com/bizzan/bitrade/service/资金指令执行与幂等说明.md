## fund 资金指令执行与幂等说明

### 1. fund 服务资金链路角色说明

- **输入 Topic**：`exchange-fund-instruction`
  - 上游 settlement 将清算结果转换为资金指令（FundInstructionDTO），按 `messageId` 发往该 Topic。
- **幂等记录表**：`processed_fund_instruction`
  - 字段核心：`message_id`、`payload`、`status(PENDING/PROCESSED/FAILED)`、时间戳等。
  - 作用：作为资金执行的“本地指令表”，记录每条 Kafka 消息的处理状态，保证幂等与可重放。
- **钱包表**：`member_wallet`
  - 字段核心：`balance`（可用）、`frozen_balance`（冻结）等。
  - 作用：用户资金汇总表，真正的余额/冻结变动都体现在这里。
- **资金流水表**：`member_transaction`
  - 用于记录充值、提现、撮合成交等资金流水，供用户前台查询。
  - 目前 fund 服务的资金指令执行逻辑 **没有直接写入 member_transaction**，而是依赖 `processed_fund_instruction` 作为“执行流水”。

### 2. FundInstructionExecuteService 设计说明

对应类：`FundInstructionExecuteService`

- **幂等主键**：Kafka `messageId`。
- **主流程（processAndExecute）**：
  1. 如果 `messageId` 已存在且 `status = PROCESSED`，直接返回，避免重复执行资金变更；
  2. 否则在 `processed_fund_instruction` 中插入/补充一条记录，状态置为 `PENDING`，保存原始 `payload`；
  3. 解析 `payload` 为 `FundInstructionDTO`，逐条执行 `instructions` 列表：
     - 调用 `MemberWalletService` 对 `member_wallet` 做加减可用、扣减冻结、解冻等操作；
  4. 全部指令执行成功：
     - 将 `status` 更新为 `PROCESSED`，记录 `processed_at`；
  5. 执行过程中抛异常：
     - 事务回滚当前资金变更；
     - 将记录标记为 `FAILED`，留给重试任务处理。
- **重试机制**：
  - `RetryFundInstructionJob` 定期扫描 `processed_fund_instruction` 中 `PENDING/FAILED` 状态的记录；
  - 调用 `retryExecutePending()` 再次执行资金指令，直到成功标记为 `PROCESSED`。

> 总体模式：本地资金指令表（processed_fund_instruction） + 钱包表（member_wallet） + 幂等 + 重试，保证 Kafka 至少一次投递下资金操作“最终只生效一次”。

### 3. executeOne 指令类型与资金含义

资金执行入口 `executeOne(FundInstructionDTO.FundInstructionItem item)` 将一条资金指令映射为一次钱包操作：

- **公共校验**
  - `amount == null` 或 `amount == 0`：直接忽略，避免无意义流水；
  - 未找到对应 `member_wallet`：抛异常中断，交由上层事务回滚。

- **INCOME**
  - 操作：`memberWalletService.increaseBalance(wallet.getId(), amount)`；
  - 含义：增加会员可用余额（例如成交买入得到的币或法币）。

- **OUTCOME**
  - 操作：`memberWalletService.decreaseFrozen(wallet.getId(), amount.abs())`；
  - 含义：从冻结余额中扣减对应金额（成交/完成订单后释放保证金）；
  - 使用 `amount.abs()` 做防御性处理，避免负数导致方向颠倒。

- **FEE**
  - 操作：`memberWalletService.deductBalance(wallet, amount.abs())`，返回值 `feeRet` ≤ 0 视为失败；
  - 含义：从会员可用余额中扣除手续费；
  - 失败处理：抛出 `IllegalStateException`，让上层回滚整笔资金指令事务。

- **FEE_REVENUE**
  - 操作：`memberWalletService.increaseBalance(wallet.getId(), amount)`；
  - 含义：平台手续费收入记账（通常 memberId 为平台账户）。

- **REFUND**
  - 操作：`memberWalletService.thawBalance(wallet, amount)`；
  - 含义：解冻余额并转为可用（撤单或部分成交后退回剩余保证金）。

- **default**
  - 未知类型仅打 `warn` 日志，不做资金变更，避免错误类型误操作资金。

### 4. fund 链路与资金流水（member_transaction）的关系

- 老逻辑（例如充值 `recharge(...)`）会在成功加余额后写一条 `MemberTransaction` 记录，作为用户可见的资金流水。
- fund 服务当前这条“清算 → 资金指令 → 钱包执行”的链路：
  - **有**：指令级别的执行流水（`processed_fund_instruction`），用于幂等与重试；
  - **有**：钱包余额/冻结的真实变动（`member_wallet`）；
  - **暂未**：同步写 `member_transaction` 的用户资金明细。

如果后续希望在前台“资金流水”中展示清算产生的资金变动，可以在 `FundInstructionExecuteService.executeOne` 中，根据指令类型补充写 `MemberTransaction`，并与当前事务保持一致。
