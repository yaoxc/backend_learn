### 1. 订单日志 + 重放整体流程

**目标**：撮合进程宕机重启后，内存中的订单簿（挂单队列 + 盘口）能恢复到宕机前的状态，且不会让已经完成/撤销的订单再次参与撮合或影响资金。

当前实现分为两块：

- **订单事件日志（OrderEventLogger）**
  - 文件路径示例：`data/wal/order-BTC-USDT.log`
  - 只记录两类事件：
    - `ORDER\t{...ExchangeOrder...}`：订单进入撮合前写入
    - `CANCEL\t{...ExchangeOrder...}`：撤单成功后写入
  - 只追加、不删除、不截断，是一个「事件流」，而不是「当前挂单快照」。

- **重启恢复（CoinTraderEvent.replayOrderLog）**
  - Spring 容器启动完成后，`CoinTraderEvent.onApplicationEvent` 会对每个 `CoinTrader` 执行：
    - 若 `match.restore.from.order.log=true`（默认）：
      - 调用 `trader.replayOrderLog()`：
        - `OrderEventLogger.replay(...)` 顺序读 `order-<symbol>.log`
        - 遇到 `ORDER` 事件 → `onOrder.accept(order)` → `trader.trade(order)`（此时为 replay 模式）
        - 遇到 `CANCEL` 事件 → `onCancel.accept(order)` → `trader.cancelOrder(order)`（replay 模式）
      - 最终效果：**把宕机前订单簿经历过的事件按顺序再“演”一遍**，恢复出与宕机前一致的挂单状态。
    - 若 `match.restore.from.order.log=false`（回退逻辑，不推荐）：
      - 从 DB 加载所有 `TRADING` 订单并 `trade(tradingOrders)`，存在重复撮合风险，仅作为回退预案。

> 关键点：日志里不只是「挂单」，还有后续的成交过程（由 replay 的 trade 流转出队）和「撤单」；  
> 回放完成后，**已完成/已撤销的订单不会留在订单簿中**。

### 2. 重放期间如何避免重复影响资金

在 `CoinTrader.trade(...)` 中有一个开关：

- `replayMode = true` 时：
  - 只做「恢复内存结构」：
    - 订单照常进入限价/市价队列，撮合逻辑会把已经完成/撤销的单从队列里移除；
  - **不对外发布撮合结果**：
    - `flushMatchResult(...)` 顶部：
      - `if (replayMode) { return; }`
    - 所以不会再次发 `exchange-trade`、`exchange-order-completed` 到 Kafka，下游资金和订单状态不会被重复更新。

整体效果：

- **订单簿恢复**：通过日志重放恢复到宕机前真实状态；
- **资金与订单终态**：由下游 DB（订单表状态 / 成交表 / 资金流水）保证，重放时不会重复改。

### 3. `seenOrderIds` 的作用与边界

在 `CoinTrader` 中新增了：

- `private final Set<String> seenOrderIds = Collections.newSetFromMap(new ConcurrentHashMap<>());`

以及调整后的 `trade(ExchangeOrder)`：

- 正常路径（`replayMode=false`）：
  - 若 `tradingHalt=true`：直接返回；
  - 若 `orderId` 已在 `seenOrderIds` 中：
    - 认为是「本进程存活期间的重复消息」（例如 Kafka/relay/outbox 重试）；
    - 打一条重复日志后直接返回，不再进入撮合；
  - 否则将 `orderId` 加入 `seenOrderIds`，再进入实际撮合逻辑。

- 重放路径（`replayMode=true`）：
  - 不使用 `seenOrderIds`，而是直接调用内部的 `internalTrade(order, checkSymbol=false)`：
    - 目的是「完全按历史事件顺序恢复订单簿」，不受本次进程中的防重集合影响。

> **定位**：`seenOrderIds` 只用于「当前进程运行期间」抵御 MQ/relay 的重复消息，  
> 不承担「记录历史所有订单」或「跨重启幂等」的职责。

### 4. 为什么重启时没必要根据订单日志恢复 `seenOrderIds`

从职责划分上看：

- **跨重启的幂等 / 一致性由谁负责？**
  - 撮合侧：订单日志重放，恢复订单簿真实状态（哪些单还挂着，哪些已完成/撤销）。
  - 下游侧（market/clearing/资金）：订单状态表 + 成交表 + 资金流水，以 `orderId`/`tradeId` + 状态机/唯一约束做幂等。

- **`seenOrderIds` 的作用范围**：
  - 只在当前 JVM 存活期间，用于避免「同一条 `exchange-order` 消息在这一轮进程中被多次处理」。
  - 进程重启后，本轮生命周期的重复消息由新的 `seenOrderIds` 从 0 开始重新防抖即可。

如果在重启时尝试「根据订单日志恢复 `seenOrderIds`」会带来两个问题：

1. **语义混淆**：  
   - `seenOrderIds` 变成了「历史所有处理过订单的全局记忆」，但它本身是一个进程内存结构，并不适合长期累积所有订单 ID。
2. **可能挡住合法补偿/重放**：  
   - 某些运维或修复场景下，需要通过 Kafka 或其他通道重新下发一笔 **仍在挂单** 的订单或某些补偿消息；
   - 如果重启后根据日志把所有曾出现过的 `orderId` 都加进 `seenOrderIds`，这些补偿消息就会被直接当成「重复」丢弃，导致补偿逻辑失效。

综上：

- **订单日志 = 事实来源（用于重启恢复）**；
- **DB 状态 = 最终裁决（资金与订单终态幂等）**；
- **`seenOrderIds` = 进程级防抖（运行期至少一次 → 逻辑上一次）**。

让这三者各司其职，比试图在重启时「恢复 `seenOrderIds`」要更简单、更安全，也更符合交易系统常见的分层设计。

### 5. 长期运行下的体量问题：`seenOrderIds` 内存 & 订单日志大小

#### 5.1 `seenOrderIds` 会不会越积越大？

会，如果用的是「无限增长」的 `Set<String>`，且撮合服务长时间不重启、订单量很大：

- 优点：本进程内重复消息几乎都能被拦截；
- 缺点：运行时间越长，集合越大，占用的内存也越多。

**推荐做法（待按需实现）：**

- 把 `seenOrderIds` 替换为带 TTL / 容量上限的本地缓存，例如：
  - Caffeine：`Cache<String, Boolean>`，设置 `maximumSize` + `expireAfterWrite`；
  - 或类似的 LRU/LFU 缓存。
- 语义上相当于：
  - 只记住「最近一段时间 / 最近 N 万条」订单 ID，用于防止 MQ 短时间重放或重试；
  - 较早的 `orderId` 被淘汰后，跨很长时间的重放依然由「订单簿当前状态 + DB 状态机」兜底。

> 这与业界常见实践一致：进程级防抖只针对“近期噪音”，不试图在内存里记住全历史。

#### 5.2 订单日志文件会不会越来越大，重放越来越慢？

当前实现下，`order-<symbol>.log` 只追加不截断，理论上会随时间线性增长：

- 对于本地开发/测试：文件很小，影响不大；
- 对于长期运行的生产环境：如果不做滚动/截断，重启时的 replay 确实会越来越慢。

**业界常见优化方向（当前文档仅做设计说明，具体实现可按需要分阶段落地）：**

- **按时间或大小滚动日志文件：**
  - 例如每天一个：`order-BTC-USDT-20260316.log`、`order-BTC-USDT-20260317.log`；
  - 或按大小拆分：每 1GB 滚一个新文件。
- **配合快照（Snapshot）：**
  - 定期对某个时间点的订单簿做一次快照（写入磁盘）；
  - 对快照时间点之前的订单日志做归档/清理；
  - 重启时：
    - 先加载最近快照恢复订单簿到某个 T0；
    - 再从 T0 之后的订单日志开始 replay，而不是从最早的那一行开始。

这样可以做到：

- **恢复正确性不变**：仍然能按顺序重放必要的订单事件；
- **恢复性能可控**：日志规模有上界，重放时间不会随着系统寿命无限增长。
