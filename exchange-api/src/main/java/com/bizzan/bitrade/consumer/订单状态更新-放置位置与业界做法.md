# 订单状态更新：放置位置与业界做法

## 一、现象与问题

撮合完成且资金入账（清算→结算→资金指令执行）均已完成，但 **订单状态仍为 TRADING，未变为 COMPLETED**。

原因：订单状态（exchange_order.status）的更新与「撮合结果」强相关，必须由**消费撮合结果**的某一方在落库时把状态从未完成改为已完成。若该消费者未部署、未消费到或未调用更新逻辑，就会出现「资金已入账但订单状态未变」。

---

## 二、订单状态应由谁、在哪里改？

### 1. 业界常见做法

| 做法 | 说明 | 适用场景 |
|------|------|----------|
| **撮合引擎内改** | 撮合引擎在内存中维护订单状态，在「完全成交」时写库更新 status | 订单表与撮合同库/同服务，强一致 |
| **撮合结果消费者改** | 撮合只发事件（如 match-result），由**拥有订单表的服务**消费事件并更新 status | 微服务拆分：撮合无库、订单服务有库 |
| **资金入账后再改** | 资金服务执行完指令后发「入账完成」事件，订单服务消费后更新 status | 要求「状态=已完成」严格等价于「钱已到账」，延迟较大 |

当前项目是 **「撮合只发事件、不写订单表」** 的架构（exchange 模块不连订单库），因此 **必须由消费「撮合结果」且能访问订单库的服务来更新订单状态**。

### 2. 本项目中「订单表」所在位置

- **exchange_order** 由 **exchange-core** 的 `ExchangeOrderService` / `ExchangeOrderRepository` 读写。
- 使用 exchange-core 的模块有：**exchange-api**（下单、查询）、**market**（行情、推送、原设计里也负责落撮合结果）。

因此，**能合法更新订单状态的只有：消费撮合结果且依赖了 exchange-core 的服务**，即 **market** 或 **exchange-api**。

### 3. 合理放置位置（本项目）

- **主路径（推荐）**：**market** 消费 `exchange-match-result`，在落「成交明细 + 订单完成」时调用 `ExchangeOrderService.processMatchResultIdempotentOrderOnly` → 内部对每个已完成订单调用 `tradeCompletedOrderOnly`，将 status 更新为 COMPLETED。
- **补充路径（已实现）**：**exchange-api** 同样消费 `exchange-match-result`（独立 groupId），只做「订单状态 + 成交明细」更新，不推行情、不碰资金。这样在**未部署 market** 或 **market 未消费到** 时，仍能保证订单状态被更新，与「清算→结算→资金」流水线一致。

两路通过 **messageId 幂等**（表 `processed_match_result_message`）去重，先处理的一方落库，后处理的一方跳过，不会重复更新。

---

## 三、已补充的操作（exchange-api 消费 match-result）

### 1. 新增逻辑

- **MatchResultOrderStatusConsumer**（exchange-api）  
  - 订阅：`exchange-match-result`  
  - GroupId：`exchange-api-order-status-group`  
  - 行为：解析 messageId、trades、completedOrders，调用 `ExchangeOrderService.processMatchResultIdempotentOrderOnly`，仅更新订单状态与成交明细，不写资金、不推行情。

- **KafkaConsumerConfiguration**（exchange-api）  
  - 提供 `kafkaListenerContainerFactory`，支持批量监听，供上述消费者使用。

### 2. 配置

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `exchange.order.status.sync.enabled` | `true` | 是否在 exchange-api 中消费 match-result 并更新订单状态。设为 `false` 则仅依赖 market 更新。 |

在 `application-dev.properties` 中已增加示例：

```properties
exchange.order.status.sync.enabled=true
```

### 3. 部署与运行要求

- **exchange-api** 需能连到与 **exchange_order** 相同的 MySQL（与 market 一致），并连到同一 Kafka 集群、能消费到 `exchange-match-result`。
- 若同时部署 **market** 与 **exchange-api**，两边都会消费同一条 match-result；幂等保证只会有一方真正落库，订单状态只会被更新一次。

---

## 四、小结

- **订单状态应放在「消费撮合结果且拥有订单表」的服务里改**；本项目即 **market** 或 **exchange-api**。
- **业界常见**：撮合只发事件，由订单域服务消费事件并更新状态；本项目采用「market 主路径 + exchange-api 补充路径」，双路幂等，避免「资金已入账但订单仍 TRADING」。
- **已补充**：exchange-api 内新增对 `exchange-match-result` 的消费，仅做订单状态与成交明细更新，可通过 `exchange.order.status.sync.enabled` 开关控制。
